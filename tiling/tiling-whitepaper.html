Concept 01 · GM 与逻辑 tensor

“一整块 tensor”不是物理 3D 内存

Ascend C 中的 GM 是扁平地址空间。Tensor 的 shape、layout、stride 决定如何解释这段地址；tiling 决定每个 core、每次循环实际处理哪一段 GM。

GM 是线性 buffer

输入 x、y 和输出 z 通常都是连续 GM 区间。可视化中的体素、网格和高亮块，只是把地址区间反解成逻辑坐标。

GM_ADDRGlobalTensor

Tensor 是解释方式

shape/layout/stride 把线性地址解释成 1D、2D、3D 或更高维逻辑空间。视图可以变，物理地址仍是线性的。

shapestridelayout

Tiling 是访问切分

每个 block 和每次 progress 只处理完整逻辑 tensor 的一个 slice。高亮区域代表当前 step 的访问范围。

blockIdxprogresstileLength

左侧是真实内存视角，右侧是调试产品视角。两者由 offset、shape、layout 和 stride 关联。
Concept 02 · Trace event 驱动高亮

3D 画面代表“当前帧”，不是静态 tiling 结果

可视化前端不应该把源码静态翻译成一张固定 3D 图。更稳妥的产品模型是：执行一次样例，记录每个 step 的 event、offset、mask、source line 和硬件路径，再用这些 trace event 驱动画面。

3D Viewport

回答“这一条 trace step 正在触碰完整 logical tensor 的哪一块”。背景是完整空间，高亮是当前访问集合。

Memory Architecture

回答“这块数据从哪里搬到哪里”。高亮 GM、UB、L1、L0、Vector、Cube 和对应 pipe。

Execution Timeline

回答“CopyIn、Compute、CopyOut、同步和 epilogue 按什么顺序发生”。点击 step 联动源码和视图。

trace event schema · conceptual
event = { phase: "CopyIn" | "Compute" | "CopyOut", op: "DataCopy" | "Add" | "EnQue" | "DeQue" | "AllocTensor", blockIdx: 0, progress: i, gmOffset: blockIdx * blockLength + progress * tileLength, range: [gmOffset, gmOffset + tileLength), sourceLine: "CopyIn(progress)", pipe: "MTE2" | "V" | "MTE3" | "Queue" }
产品判断：3D tensor 不表达硬件阶段。AIC/AIV handoff、SetFlag/WaitFlag、pipe bubble 应放进 timeline 和 memory architecture；3D 只负责“在哪一块算”。
Concept 03 · Tiling 公式

Tiling 最终落到 GM offset 和 length

以 Vector Add 为例，x、y、z 都是 GM 上的扁平 buffer。Tiling 需要决定每个 core 的 blockLength、每次循环的 tileLength，以及当前 progress 对应的 gmOffset。

blockLength totalLength / blockNum 每个 core 负责的完整逻辑区间长度
tileLength blockLength / tileNum / BUFFER_NUM 一次循环处理的 tile 长度，双缓冲会影响单 slot 容量
gmOffset blockIdx * blockLength + progress * tileLength 当前 core、当前循环访问的 GM 起点
Vector Add · current loop slice
xGM[gmOffset : gmOffset + tileLength] yGM[gmOffset : gmOffset + tileLength] zGM[gmOffset : gmOffset + tileLength]

源码里的 i 是 tile progress

for (int i = 0; i < loopCount; i++) 中的 i 通常不是随意计数器，而是当前 core 内的 tile progress。点击第 i 次循环，3D 高亮就应该跳到对应 slice。

尾块和 mask 应该显式展示

当 totalLength 不能整除 blockNum、tileNum 或向量对齐长度时，尾块、padding、mask 是最容易误解的调试点，应该在 Inspector 中显示有效元素数。

Concept 04 · 1D 数据如何画成 3D

3D 是执行结构投影，不是原始 tensor 物理形状

Vector Add 的输入可能只有 1D，但为了教学和调试，可以把访问空间投影到三维：X 表示 tile 内元素，Y 表示当前 core 内的 progress，Z 表示 block/core。

X axis element inside tile tile 内第几个元素，用于观察 vector lane、对齐和 mask。
Y axis tileIdx / progress 当前 core 内第几次循环，用于观察 loop schedule。
Z axis blockIdx / core 不同 core 负责的 block，用于观察并行切分。
1D Vector Add 的 3D 调试投影
这张图的价值不是还原物理内存，而是让开发者看到“哪些 core 切了哪些 tile、当前 step 在整体空间中落在哪里”。
Concept 05 · 官方图可补充的具体指令信息

Vector Add 可以拆成三段指令队列

用户给出的两张官方图最有价值的信息，是把 CopyIn、Compute、CopyOut 从抽象阶段拆成了可落到 Inspector 和 timeline 的具体 API 调用。下面按图中的 16 条序列整理。

CopyIn · 搬入单元指令队列

  1. 1xl = x.AllocTensor()
  2. 2yl = y.AllocTensor()
  3. 3DataCopy(xl)
  4. 4DataCopy(yl)
  5. 5x.EnQue(xl)
  6. 6y.EnQue(yl)

核心含义：从 GM 读取 x/y 当前 slice，经 MTE2 搬入 Local Memory VECIN，并把 local tensor 放入输入队列。

Compute · 计算单元指令队列

  1. 7xl = x.DeQue()
  2. 8yl = y.DeQue()
  3. 9zl = z.AllocTensor()
  4. 10Add(zl, xl, yl)
  5. 11x.FreeTensor(xl)
  6. 12y.FreeTensor(yl)
  7. 13z.EnQue(zl)

核心含义：从输入队列取 x/y，在 Vector pipeline 执行 Add，结果 zLocal 入输出队列。

CopyOut · 搬出单元指令队列

  1. 14zl = z.DeQue()
  2. 15DataCopy(zl)
  3. 16z.FreeTensor(zl)

核心含义：从 VECOUT 队列取 zLocal，经 MTE3 写回 GM 的 z 当前 slice，并释放本地 tensor。

三段指令队列 · Queue / Pipe / Local Memory 的对应关系
青色主线是数据流，红色虚线是队列与 pipe 之间的同步关系。官方图里的 SetFlag/WaitFlag 不一定直接出现在用户代码里，但它们解释了 queue 为什么能跨搬运单元、计算单元、搬出单元协调。

Concept 06 · Double Buffer 与流水并行

MVP 可以串行播放，增强版要显示三条泳道

BUFFER_NUM = 2 的直觉是让搬入、计算、搬出尽量重叠。概念上，在同一时刻可能 CopyIn tile i+1、Compute tile i、CopyOut tile i-1。

MTE2 / CopyIn lane
tile i tile i+1 tile i+2
Vector / Compute lane
tile i tile i+1 tile i+2
MTE3 / CopyOut lane
tile i tile i+1

顺序播放用于入门

MVP 可以先展示 CopyIn(i) → Compute(i) → CopyOut(i)，让用户建立 tile 生命周期。

三泳道用于调优

增强版应显示 MTE2、Vector/Cube、MTE3 三条 lane，解释 pipe bubble、搬运等待和计算等待。

队列状态是关键

只有看到 EnQue/DeQue、buffer slot 和 wait reason，用户才能判断 double buffer 是否真正隐藏了延迟。

Concept 07 · MatMul 迁移

MatMul tiling 切的是 M/N/K，不只是 1D slice

MatMul 的逻辑公式是 C[M,N] = A[M,K] × B[K,N]。3D 视图可以把 X/Y/Z 映射为 N tile、M tile 和 K reduction step。

A tile

A[mTile, kTile] → L0A。3D 视口高亮当前 M/K 范围，memory architecture 高亮 L1/L0A 路径。

B tile

B[kTile, nTile] → L0B。和 A 共同决定一次 Cube compute 的输入窗口。

C tile

Cube → L0C[mTile,nTile] → GM:C。C 本身通常是 2D，K 是 reduction 过程维度。

轴Vector Add 映射MatMul 映射读者应看到什么
Xtile 内元素N tile输出列方向的切分
YprogressM tile输出行方向的切分
ZblockIdx/coreK tile / reduction step累加过程，而非输出 tensor 物理第三维
迁移原则：3D 视图继续回答“在哪一块算”；Memory Architecture 才回答 “A/B/C 经过 L1、L0A、L0B、L0C、Cube 的哪条路径”。
Concept 08 · Fusion 关注点

Fusion 的重点是中间结果是否少走 GM

融合算子的价值不只是把激活函数也画出来，而是要显示中间结果有没有绕回 GM workspace。AIC/AIV handoff 应放进 timeline 和 memory architecture，不要硬塞进 tensor 轴。

Terms · 术语表

页面中必须稳定使用的术语

术语含义在可视化里怎么表达
GMGlobal Memory，全局线性地址空间。完整 tensor 数据的物理存放位置，3D 中只显示逻辑解释。
Logical tensor space由 shape/layout/stride 对 GM 地址的逻辑解释。3D 视口背景网格。
blockIdx当前运行的 core/block 标识。可映射到 3D Z 轴或 MatMul 的分块维度。
progress当前 core 内的 tile 循环进度。可映射到 3D Y 轴，并驱动当前高亮 slice。
gmOffset当前 tile 在 GM 中的起始 offset。Inspector 中的核心字段，3D 高亮由它反解。
TPipe管理本地内存、队列和事件的 pipe 对象。显示 queue、buffer slot、SetFlag/WaitFlag 的同步语义。
TQue跨搬运单元、计算单元传递 LocalTensor 的队列。显示 EnQue/DeQue、queue depth、producer/consumer。
VECIN / VECOUTVector pipeline 的输入/输出 Local Memory 队列区域。Memory Architecture 中的本地 buffer。
MTE2 / MTE3常用于 GM → Local 和 Local → GM 的搬运 pipeline。timeline lane 与 architecture path 高亮。
Vector / CubeAI Core 上的计算 pipeline。Add、Mul 等 Vector 指令或 MatMul Cube 指令高亮。
BUFFER_NUM本地队列的 buffer slot 数，常见为 2。双缓冲时显示 slot 轮换和多泳道重叠。
Fusion把多个阶段合并，减少中间结果回 GM 的机会。高亮 C-V direct route，并标记 GM workspace avoided。
