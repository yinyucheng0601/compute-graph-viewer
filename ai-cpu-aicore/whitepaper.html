Concept 01 · 角色分工

把昇腾算子系统当成一个产品系统看

五个角色各司其职：CPU 侧做编排、AI Core 做高吞吐计算、内存与调度资源做协同载体。把它类比成一套 App 服务体系，分工立刻清晰。

角色产品类比主要职责代码里的典型位置
Host CPU
 App 服务端 / 调用入口 Python、PyTorch、ACL 发起调用，管理 device、stream、内存 torch_ops_extension/custom_ops/csrc/*.cpp、示例里的 aclrt*
AI CPU
 设备侧调度 / 控制协处理器 部分 shape/tiling/控制类任务、AI CPU 算子、复杂图或 PTO/KFC 类编排 op_host/*_tiling.cpp、PTO 里的 SetAttachedStreamInfos
AI Core
 执行引擎 运行 __aicore__ kernel，执行 vector / cube / data-copy 指令 op_kernel/*.cpp、.asc kernel
GM
 共享货架 输入输出 tensor、workspace、tiling 数据所在的全局内存 GM_ADDR、GlobalTensor
UB / L1 / L0
 AI Core 内部工作台 单核内部的搬入、计算、搬出 TPipe、TQue、LocalTensor、__ubuf__
关键边界：CPU 侧（Host CPU / AI CPU / CANN Runtime）做编排——识别算子、检查 shape/dtype、选实现、算 tiling、备 workspace、生成任务描述并投递到 stream；AI Core 做高吞吐计算——读输入 GM 地址与 tiling，按 block/task 并行执行已编译好的 kernel 指令。
Concept 02 · 端到端链路

一次算子调用，从 Python 走到硬件流水

最关键的一点：AI CPU/Host CPU 准备“任务和参数”，AI Core 执行“已经编译成设备指令的 kernel”。CPU 侧不会在运行时逐元素指挥 AI Core。

端到端链路 · 控制面下单 → 数据面执行
蓝线是控制面调用链（Python → C++ → Runtime → CPU 控制侧）；橙色主线是 CPU 把“订单”写入共享内存；绿色是 AI Core 在 stream 上被拉起后独立执行 kernel。CPU 侧不会进入绿色框逐条指挥。
两层“指令”要分清：调度指令（Runtime/CPU 侧可见）说的是“启动哪个 kernel、多少核、在哪个 stream、依赖谁”；AI Core 指令（设备 kernel 内部）才是“MTE2 搬入、Vector/Cube 计算、MTE3 搬出、SetFlag/WaitFlag 同步”。
Concept 03 · 代码结构

一个算子，在代码里被拆成三层

在 cann-recipes-infer-master/ops/ascendc 下，一个融合算子通常拆成 op_host（控制面）、op_kernel（数据面）和 torch_ops_extension（框架接入）三块。这不是炫技，而是同时服务易用性、正确性和性能。

Concept 04 · 传递内容

AI CPU 到 AI Core，到底传了什么

CPU 侧没有直接传“VADD 指令”给 AI Core。它传的是一份 kernel 任务的“订单”——七类参数共同描述这次任务该怎么跑。AI Core 拿到订单后，按 kernel 代码和 tiling 参数独立执行。

传递内容作用代码信号
kernel 选择选哪个 op、哪个 SoC、哪个 binary / 实现AICore().AddConfig(...)、OPP 包
blockDim启动多少个 AI Core blockSetBlockDim(...)、kernel 里 GetBlockIdx()
tilingKey选择 kernel 内哪条实现分支SetTilingKey(...)、TILING_KEY_IS(...)
tilingData每核 / 每循环 / 每 tile 的切分参数SaveToBuffer(...)、GET_TILING_DATA(...)
workspace size临时内存需求GetWorkspaceSizes(...)
tensor 地址输入输出的 GM 地址GM_ADDR、GlobalTensor.SetGlobalBuffer(...)
stream / event / notify调度顺序和跨任务同步aclrtStream、SetAttachedStreamInfos / SetSyncResInfos
把它当一张订单看：kernel 选择 = 选哪条产线，blockDim = 开几条流水线，tilingKey = 走哪个工艺分支，tilingData = 每批做多少，workspace = 备料区，tensor 地址 = 原料和成品货位，stream/event = 排产与交接顺序。
Concept 05 · 硬件流水

AI Core 内部真正执行的是什么指令

asc-devkit 文档把 AI Core 上的流水分成多类。C API 是指令层 API，直接对应一条或多条硬件指令——asc_copy_gm2ub_sync 走 MTE2、asc_add_sync 走 V、asc_copy_ub2gm_sync 走 MTE3。

PIPE_S
标量 / 控制流水 · 标量计算、控制类操作
PIPE_V
Vector 流水 · Add / Mul / Exp / Reduce
PIPE_M
Cube / 矩阵流水 · Matmul / MMAD
PIPE_MTE2
搬入流水 · GM → UB / L1 / L0
PIPE_MTE3
搬出流水 · UB / L1 → GM
PIPE_MTE1
内部搬运 · L1 → L0A / L0B / UB
PIPE_FIX
Fixpipe · Cube 结果搬出 / 格式处理
计算流水
S / V / M / FIX 负责“算”
搬运流水
MTE1 / 2 / 3 负责“搬”

AI Core 高性能来自三件事

多核并行

blockDim 决定启动多少个 block，kernel 内用 GetBlockIdx() 找到当前核处理的数据范围。

核内流水并行

CopyIn、Compute、CopyOut 可分别走 MTE2、V/Cube、MTE3，互不阻塞地流水起来。

局部内存复用

数据从 GM 搬到 UB / L1 / L0 后，在 AI Core 内部高带宽处理，最后再写回 GM。

开发者写的是抽象形式：DataCopy、Add、TPipe/TQue、asc_add、asc_sync。编译器和 CANN 工具链把它们变成 AI Core 能执行的设备指令。
Concept 06 · 最小心智模型

CopyIn → Compute → CopyOut

Ascend C 入门 Add 示例的核心逻辑就是这条三步链。绝大多数 AI Core 算子都是它的扩展——复杂算子只是把它扩成更多 tile、更多 pipeline、更多分支、更多临时 buffer。

AI Core 算子底层套路
Global Memory 输入 x / y | | CopyIn : MTE2 搬入 v Local Memory / UB | | Compute : Vector 或 Cube 计算 v Local Memory / UB | | CopyOut : MTE3 搬出 v Global Memory 输出 z
Tiling 解决的核心体验问题：用户希望“输入一个大 tensor、拿到一个结果”；硬件实际需要把大 tensor 拆成很多小块，让多个 AI Core 并行搬、算、写。Tiling 就是把“用户视角的大任务”转成“硬件视角的可执行任务包”。
Concept 07 · 控制面定位

AI CPU 是控制面，不是主算力面

初学者容易把 AI CPU 理解成“AI Core 的老板，实时下发每条计算指令”。这不准确。更好的说法是：AI CPU 偏控制面，AI Core 偏数据面。

AI CPU 偏控制面适合分支多、控制复杂、shape / tiling / 同步 / 调度类工作，或执行 AI CPU 算子。
AI Core 偏数据面适合高吞吐、规则化、批量的 tensor 计算。核心向量 / 矩阵算力都落在这里。
分工是工具链层面的opbuild 提供 --aicpu、--aicore、--hostcpu 三种生成模式，CANN 把它们视为不同执行后端。

op_proto_generator.cpp 里出现了 TILING_ON_HOST 和 TILING_ON_AICPU 的 placement 生成逻辑——说明 tiling 可以有 Host / AI CPU 侧的位置差异。普通融合算子里 tiling 常在 CPU 侧准备参数；图模式或设备侧编排场景下，tiling/control 可能放到 AI CPU 上。

PTO / KFC：AI CPU 参与编排的更明显信号

ops/pypto 的 PTO 算子比普通 Ascend C 算子更能看到 AI CPU 参与控制的痕迹。例如 lightning_indexer_pto/op_host/lightning_indexer_proto.cpp 里设置了 attached stream 名称 pto aicpu kfc server、reuse key pto kfc_stream、sync resource SYNC_RES_NOTIFY。

普通算子 vs PTO/KFC 类复杂图算子
普通 AI Core 算子： CPU 侧算好 tiling ──▶ stream 上启动 AI Core kernel ──▶ AI Core 写回输出 PTO / KFC 类复杂图算子： 需要一个设备侧控制服务 / 附加 stream 协调多个 kernel、同步资源、动态 tile 或多阶段执行 → AI CPU 更像“设备侧执行协调员”
边界要记住：这类场景中 AI CPU 参与调度和同步，但核心矩阵 / 向量吞吐仍主要落在 AI Core kernel 上。它是协调员，不是算力主力。

一个实用判断：这段代码属于控制面还是数据面

你看到的代码多半属于

初学者最容易混淆的 8 个点

1 · op_host ≠ 独占某 CPU

它是“算子控制面代码”的目录名，运行位置取决于图执行、tiling placement 和算子类型。

2 · AI CPU 不做高吞吐计算

核心逻辑是大规模向量 / 矩阵计算时，目标应是 AI Core。

3 · AddConfig 是声明后端

AICore().AddConfig(...) 告诉 CANN 这个 op 有 AI Core 实现并支持哪些 SoC。

4 · blockDim ≠ tensor shape

shape 是用户数据形状，blockDim 是这次任务启用多少个 AI Core block。

5 · tilingKey 是分支选择器

常用来区分 normal、single row、slice D 等实现路径。

6 · GM 不是高速工作区

GM 是全局内存；高性能通常要求把数据搬到 UB / L1 / L0 再算。

7 · 同步不是默认全自动

TQue/TPipe 简化队列同步；低层或手写 pipeline 仍要懂 MTE2_V、V_MTE3、PipeBarrier。

8 · 框架调通 ≠ kernel 高效

PyTorch 自定义 op 和 AI Core kernel 是两层交付物；kernel 高效还需注册、converter、安装 OPP/whl。

Findings · 最终判断

结论与给初学者的行动建议

同一个算子被拆成很多层，不是复杂炫技，而是 CANN/Ascend C 在同时平衡三类用户诉求：算法开发者想快、算子开发者想稳、性能工程师想榨干硬件。

核心张力用户要的是“一个大 tensor 进、一个结果出”；硬件要的是切块、多核并行、局部内存复用。中间靠 tiling 与控制面把两种视角对齐。
优化方向先建立“控制面下单、数据面执行”的分工认知，再按 docs → csrc → def → tiling → kernel 的顺序读码，最后才进 kernel 性能优化。
局限性AI CPU 始终是控制 / 协调角色，无法替代 AI Core 承担高吞吐算力；PTO/KFC 编排能力强，但矩阵向量吞吐仍压在 AI Core 上。

Product Definition · 指令可视化系统

把心智模型做成一个可操作的工作台

前面的白皮书解决“AI CPU 与 AI Core 怎么协同”的认知问题；产品化的下一步，是把这套认知转成一个联动可视化系统，让用户从 kernel 源码出发，看见控制面任务订单、AI Core 指令流水和芯片架构路径之间的关系。

产品定义：指令可视化系统是一个面向昇腾算子开发与性能优化的工作台。它把 kernel 源代码、CANN 控制面任务参数、AI Core 指令流水、芯片架构图放到同一个界面里，让用户点击一行代码，就能同步看到这行代码落在哪条硬件路径、使用哪些流水单元、依赖哪些 tiling / stream / GM / workspace 参数，以及下一步应该如何优化。

产品目标

在 30 秒内理解一个 kernel 的关键执行路径，在 2 分钟内从一行源码定位到对应硬件路径和性能风险，在 10 分钟内完成一次初步瓶颈判断。

业务目标

降低 Ascend C / CANN 算子开发入门门槛，提升 910B 到 950 / 950B 迁移解释能力，把 kernel 优化从专家经验推进到源码、架构、tiling、pipeline 的证据联动。

产品原则

源码是入口，架构图常驻，联动优先于堆指标；控制面和数据面必须同时可见，优化建议必须先给解释和证据。

Product Scenarios · 核心使用场景

用户不是来看架构图，而是来回答具体问题

指令可视化系统要服务的是 kernel 研发和调优现场。每个场景都必须把“用户要问的问题”落到源码、控制面参数、硬件路径和 Inspector 证据上。

S1 源码到指令路径点击 DataCopy、Add、Mmad、PipeBarrier，看到 memory / compute / control 分类和对应流水。
S2 控制面任务订单从 SetBlockDim、SetTilingKey、workspace、stream、event / notify 理解 CPU 侧传给 AI Core 的“订单”。
S3 性能结构诊断围绕 CopyIn → Compute → CopyOut 查看 tile、block、GM / UB / L1 / L0 数据路径和 pipeline bubble。
S4 910B / 950 对比同一行源码在两代芯片上走不同 selectors / routes，Inspector 给出 cycle 差异和路径退化说明。
S5 PTO / KFC 编排识别 attached stream、sync resource、notify，把它们归入设备侧控制编排，而不是普通 AI Core 计算。
S6 Review 报告导出源码片段、硬件路径、tiling、pipeline、建议和证据，供 code review 或教学复盘。
场景边界：这个系统不替代 IDE，也不在 P0 自动改代码。它负责把“为什么这样跑、瓶颈在哪里、下一步怎么查”解释清楚，并把证据留给 reviewer。
Product Shape · 三栏联动工作台

首屏必须是可操作工作台，不是说明页

MVP Scope · 功能范围与数据契约

P0 先打通一条可信链路

P0 的目标不是做全自动优化器，而是让用户点击关键源码行时，能够看到硬件路径、指令流水、任务订单、tiling、pipeline 和证据。只要这条链路可信，后续 profile、编译产物、IDE 插件才能接进来。

功能描述验收标准
Kernel 项目载入载入一个 kernel 示例，包含源码、架构 preset、标注数据。可打开 Vector Add、GELU、FP8 MMAD 三类示例。
源码行级标注标出 memory、compute、control、scalar、loop。关键行点击后 Inspector 与硬件图同步刷新。
硬件路径高亮根据 selectors 和 routes 高亮架构图节点与连线。点击 DataCopy 能看到 GM / L2 / UB 或 L1 路径。
指令解释卡展示当前行的语义、流水、原因、建议。每个关键行必须有一句中文解释和至少一个 evidence。
控制面任务订单汇总 kernel 选择、blockDim、tilingKey、tilingData、workspace、stream / event。op_host 与 launch 行能被归入任务订单。
Tiling 可视化展示 tile shape、tile count、per-block range、片上内存占用。调整主 tile 参数时，下游卡片刷新。
Pipeline 泳道图展示 MTE / Vector / Cube / Scalar / FixPipe 时间线。能标出 critical path 和 bubble。
910B / 950 对比同一源码行在两代芯片中映射不同路径。Compare 模式有一句明确差异结论。

Metrics · 成功指标与验收

产品是否有用，取决于解释是否能被复核

指令可视化系统的成功不只是“图能亮”，而是用户能更快完成一次可信的 kernel 判断，并且每条建议都能回溯到源码、控制面参数或硬件证据。

30s用户导入 kernel 后看到第一条源码到硬件路径联动
10min新用户能说清 control plane / data plane 分工并完成初步诊断
90%P0 示例 kernel 的 memory / compute / control 关键行覆盖率

P0 验收清单

验收项通过标准
首屏形态打开产品后，同时看到源码、芯片架构图、Inspector。
访存路径联动点击 DataCopy 行，硬件图高亮 GM / L2 / UB 或 L1 路径，Inspector 显示 MTE2 / MTE3 解释。
计算路径联动点击 Add / Mmad 行，硬件图高亮 Vector 或 Cube 路径，Inspector 显示 compute pipeline。
控制面识别点击 SetBlockDim / SetTilingKey 行，Inspector 将其归入控制面任务订单，而不是 AI Core 计算指令。
芯片对比Compare 模式能展示同一行在 910B 与 950 的路径差异，并给出一句明确差异结论。
证据完整每个关键行至少有 kind、tag、path、pipeline、evidence。
空态清晰无路径、无 profile 或架构节点缺失时，有明确说明，不出现空白面板。
非目标P0 不做真实 profile 采集，不自动改写 kernel，不替代 IDE，也不做完整 CANN 文档搜索系统。
主要风险高阶 API、宏、模板、自定义封装可能让静态解析漏判；profile 缺失时只能给 source-inferred 估算。
产品结论它不是把白皮书做成网页，而是把“控制面下单、数据面执行”的洞察做成可交互的工程工具。
Appendix · 术语表

术语表

报告中出现的关键术语、定义与首次出现的楼层。专有名词保留英文。

术语定义位置
Host CPU发起调用的主机 CPU，运行 Python / PyTorch / ACL，管理 device、stream、内存五个角色
AI CPU设备侧调度 / 控制协处理器，承担部分 shape/tiling/同步/调度类任务及 AI CPU 算子五个角色 / 控制面
AI Core设备执行引擎，运行 __aicore__ kernel，做高吞吐向量 / 矩阵计算五个角色
CANN Runtime / GE负责查 OPP 信息库、InferShape、Tiling、选 kernel 并在 stream 上下发任务的运行时端到端链路
GMGlobal Memory，全局内存，存放输入输出 tensor、workspace、tiling 数据五个角色
UB / L1 / L0AI Core 内部的局部高带宽工作区，单核搬入、计算、搬出在此完成五个角色
op_host算子控制面代码目录：注册、InferShape、Tiling、平台信息读取三层结构
op_kernel算子数据面代码目录：AI Core kernel 的真正计算实现三层结构
Tiling把用户视角的大 tensor 任务切成硬件可执行的小任务包的过程心智模型
blockDim本次任务启动的 AI Core block 数量，即并行度（≠ tensor shape）传了什么
tilingKeykernel 内部实现分支选择器，区分 normal / single row / slice D 等路径传了什么
tilingData每核 / 每循环 / 每 tile 的切分参数，存入 raw tiling buffer传了什么
workspacekernel 执行所需的临时内存，大小由 GetWorkspaceSizes 给出传了什么
stream任务队列，决定 kernel 任务的调度顺序端到端链路
event / notify跨任务同步资源，PTO 中以 SYNC_RES_NOTIFY 等形式出现控制面
PIPE_V / PIPE_M / PIPE_SAI Core 上的 Vector / Cube 矩阵 / 标量计算流水指令流水
MTE1 / MTE2 / MTE3AI Core 上的内部搬运 / 搬入（GM→UB）/ 搬出（UB→GM）流水指令流水
CopyIn / Compute / CopyOutAI Core 算子的最小三步链：搬入 → 计算 → 搬出心智模型
OPP算子信息库 / 算子包，CANN 据此查找算子注册信息与 binary 实现端到端链路
PTO / KFC需要设备侧控制服务、附加 stream 协调多 kernel 的复杂图算子编排场景控制面
ACLNNCANN 提供的算子调用封装层，EXEC_NPU_CMD_V1(aclnnXxx, ...) 经此发起端到端链路
convertertorch.compile / 图模式下把 PyTorch op 转成 GE custom_op 的转换器三层结构
Instruction Visualizer指令路径工作台，把 kernel 源码、控制面任务参数、AI Core 指令流水和芯片架构图联动展示产品化
Inspector右侧解释面板，围绕当前源码行或当前 kernel 展示硬件路径、指令流水、任务订单、tiling、pipeline、建议和证据工作台
selectors / routes源码行标注映射到架构图的节点选择器和路径 id，是源码与硬件图联动的稳定契约联动模型
source-inferred仅基于源码和静态规则推断的解释层级；无 profile 数据时可以使用，但不能给强性能结论指标与验收
