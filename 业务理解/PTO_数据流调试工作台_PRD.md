# PTO 数据流调试工作台 PRD

**版本**：v0.1  
**状态**：草案  
**面向对象**：产品、设计、前端、可视化研发  
**一句话定义**：PTO 不是再做一个图查看器，而是做一个让开发者“跟着一条数据走完整个旅程”的调试工作台。

---

## 一、产品定位

PTO 要解决的不是“图画得够不够全”，而是“开发者能不能围绕一个具体问题，快速找到足够的证据”。

在 PyPTO / CANN / DeepSeek 这类场景里，用户真正关心的问题通常不是：

- 这个模块叫什么
- 这个 Pass 有多少节点
- 这段源码长什么样

而是：

- 这个 tensor 从哪来
- 为什么会分成这些 path
- 哪一步开始值不对
- 为什么这次性能变差

因此，PTO 的第一组织原则不应该是页面模块，也不应该是编译阶段，而应该是**一条数据的旅程**。模型层、源码层、编译层、执行层，都是这条旅程的不同观察镜头。

---

## 二、目标用户

### 2.1 核心用户

PyPTO / Ascend 算子开发者。  
他们已经有源码、Pass 产物、controlflow、泳道图等材料，但缺少一个能把这些材料串起来的工作台。

### 2.2 次级用户

性能分析工程师、编译器工程师，以及需要快速理解问题上下文的产品经理或新同学。

### 2.3 不作为首版核心服务对象的人群

纯模型算法研究者、只关心整网结构的架构评审人员、完全没有 PyPTO 背景的外部用户。首版不为他们设计。

---

## 三、要解决的核心问题

首版只解决 3 类高频问题。

### 3.1 理解问题

“这个计算在模型里是什么，在源码里怎么写，在编译后变成了什么？”

### 3.2 正确性问题

“值从哪一步开始不对？是 loop 边界、shape 绑定、path 分叉，还是中间 tensor 出现异常？”

### 3.3 性能问题

“为什么这次慢了？是 tiling 变了、子图切分变了，还是执行路径和调度变了？”

---

## 四、产品目标

### 4.1 目标

让用户从一个具体的 tensor、kernel 或 path 出发，在同一个工作台里完成下面这条旅程：

```text
定位入口
-> 看清当前数据对象
-> 沿模型/源码/编译三个层次下钻
-> 对比正常与异常或不同 profile
-> 得出可以行动的结论
```

### 4.2 非目标

首版不做下面这些事：

- 不做通用大模型可视化平台
- 不做任意格式文件的万能图浏览器
- 不做自动修复或自动优化建议系统
- 不做完整训练期 debugger

---

## 五、核心体验定义

### 5.1 体验原则

第一，用户从“问题”进入，而不是从“页面”进入。  
第二，用户在同一条旅程里缩放，而不是在三个独立工具之间跳转。  
第三，页面先给证据，再给概念解释。  
第四，对比是首版一等公民，因为很多真实问题只有在对照下才看得清。

### 5.2 首版体验主线

用户打开 PTO 后，不是先选 `pass-ir/index.html / source-flow/index.html / model-architecture/index.html` 这样的模块，而是先选一个问题入口：

- 查看一个 kernel 的数据旅程
- 对比两个 profile / 两次编译结果
- 定位某个 path 为什么出现
- 分析某个 tensor 为什么异常

进入工作台后，页面结构固定为一套：

- 顶部：当前问题上下文与路径面包屑
- 左侧：旅程导航，按模型层 / 源码层 / 编译层组织
- 中间：主画布，展示当前层级下的数据流图
- 右侧：证据面板，展示 shape、dtype、源码位置、path、tile、异常标记、对比结果
- 底部：时间/阶段轨道，用于查看 Pass 演进、path 演进、profile 差异

这套结构不因为切到不同层就完全换页，只是主画布和右侧证据内容发生变化。

### 5.3 一个典型使用过程

以“为什么 `Indexer Prolog` 会分成 6 条 path”为例，用户的体验应是：

1. 从问题入口选择 `Indexer Prolog · loop_unroll 多 path`
2. 首屏直接看到当前 kernel 的摘要：输入 shape、`unroll_list`、对应模型位置
3. 在源码层看到 `t_idx` loop 与 `loop_unroll` 定义
4. 下钻到编译层，看到 `step -> path` 映射和 6 条 path 的出现原因
5. 点击任一路 path，右侧显示对应的 step、tile、Before/After Pass 差异
6. 用户能够回答：“这些 path 不是随机生成的，而是由 `unroll_list=[32,16,8,4,2,1]` 推出来的”

如果 PTO 不能支持用户完成这 6 步，这个体验就还没有成立。

## 六、信息架构

### 6.1 入口页

入口页不是“模块选择器”，而是“问题入口页”。

首版提供 3 个入口：

- 理解一条数据旅程
- 对比两个结果
- 定位异常或性能问题

入口页下方提供“首批内置案例”，让用户直接进入，不要求先准备所有文件。

### 6.2 工作台

工作台是 PTO 的核心页面。

#### A. 上下文栏

展示当前对象：

- 模型位置
- kernel 名称
- 当前 case 名称
- 当前 shape/profile
- 当前层级

#### B. 旅程导航

不是文件树，而是数据旅程树：

- 模型层
- 源码层
- 编译层
- 对比结果

每一层都能显示“当前是否有异常、是否有差异、是否有可继续下钻的证据”。

#### C. 主画布

同一块画布，支持 3 种观察模式：

- 语义流视图：强调“这个 tensor 经过了哪些语义步骤”
- 源码控制流视图：强调 loop、分支、循环体
- 编译执行视图：强调 path、tile、subgraph、execute graph

这三种不是三个页面，而是同一画布上的不同镜头。

#### D. 证据面板

右侧面板必须是首版重点，不是附属详情。

它至少要能稳定展示：

- tensor 名称、shape、dtype、来源
- 对应源码文件和关键行
- loop 信息、`unroll_list`、当前 step
- path magic、Before/After Pass 对照
- tile 配置和关键执行参数
- 是否命中异常规则或差异规则

#### E. 对比轨道

底部轨道首版至少支持两类对比：

- Pass 演进对比
- 两个 profile / 两次编译结果对比

## 七、功能需求

### 7.1 问题入口

系统需要支持以下入口方式：

- 从内置 case 进入
- 从 kernel 名称进入
- 从某个 path 文件进入
- 从某个 pass 产物进入

首版优先做“从内置 case 进入”和“从 kernel 进入”。

### 7.2 数据旅程缩放

用户应能在同一条旅程中完成：

- 从模型层下钻到源码层
- 从源码层下钻到编译层
- 从编译层返回到语义层

切换过程不能跳到完全不同的页面，也不能丢失当前上下文。

### 7.3 证据关联

任一节点或边被点击后，系统要能联动展示：

- 对应源码位置
- 对应 path / tile / pass 信息
- 对应输入输出 tensor 信息

如果某一层缺失映射关系，页面要明确说明“当前没有证据”，而不是静默失联。

### 7.4 对比模式

系统需要支持以下两种对比：

- 正常 vs 异常
- profile A vs profile B

对比时，页面要直接回答“哪里变了”，而不是让用户自己并排肉眼找。

### 7.5 用例驱动内容

首版不是做“任意图文件浏览器”，而是做“问题驱动的案例工作台”。

因此内置 case 不是演示素材，而是产品内容本身。每个 case 都必须带：

- 用户问题
- 入口对象
- 相关源码
- 相关 Pass 产物
- 预期用户结论

## 八、首版用例池

首版建议先做 6 个内置 case。它们不是 QA 测试集，而是产品体验内容。

### Case 1：Indexer Prolog 小图正常链路

目标：让用户第一次看懂“同一条数据在三个层级里怎样对应”。

输入建议：

- `models/deepseek_v32_exp/lightning_indexer_prolog_quant_impl.py`
- `output_deepseek/Pass_00_RemoveRedundantReshape`
- `output_deepseek/kernel_aicpu/controlFlow_host_*.cpp`

### Case 2：loop_unroll 生成多 path

目标：让用户理解 `unroll_list` 如何映射成多条 path。

输入建议：

- `deepseekv32_lightning_indexer_prolog_quant.py`
- `PATH0_6/8/10/12/14/16` 相关产物

### Case 3：Sparse Flash Attention 的 tile loop

目标：让用户理解“tile 遍历本身就是算法的一部分”。

输入建议：

- `sparse_flash_attention_quant_impl.py`
- 对应 controlflow 和编译产物

### Case 4：dynamic shape 绑定

目标：让用户看懂“符号维度怎样绑定成实际维度”。

输入建议：

- `python/tests/ut/ds_v32/test_dynamic_mla_prolog.py`
- 相关动态 shape 产物

### Case 5：两套 tiling / 两套 profile 对比

目标：让用户看懂“性能变化到底是怎么来的”。

输入建议：

- DeepSeek 两套 profile 配置
- 对应 Pass / path / subgraph 差异

### Case 6：正常 vs 异常

目标：验证 PTO 是否真的能帮助用户定位问题，而不只是展示结构。

输入建议：

- 一个已知正确 case
- 一个已知异常 case
- 两者对照的关键证据

## 九、数据需求

PTO 首版不需要海量线上数据，但需要一套结构完整的“问题包”。

每个问题包至少包括：

- case 基本信息
- 问题描述
- 模型层位置
- 源码文件与关键行
- pass 产物
- controlflow / execute 相关产物
- profile / shape 信息
- 预期结论

从现有材料看，首版数据源已经基本具备：

- `models/deepseek_v32_exp`
- `python/tests/ut/ds_v32`
- `output_deepseek`

缺的不是数据量，而是把这些数据重新组织成问题包。

## 十、成功标准

首版不以“支持多少文件格式”为成功标准，而以“用户能不能回答问题”为成功标准。

### 10.1 理解效率

新用户在 5 分钟内，能用一个内置 case 回答：

- 当前 kernel 在模型里做什么
- 这条数据在源码里怎么被处理
- 为什么会出现当前这些 path

### 10.2 定位效率

用户在 10 分钟内，能基于一个对比 case 明确说出：

- 差异出现在什么层
- 差异更像源码问题、编译问题，还是执行问题

### 10.3 证据完整度

每个首版 case 都能提供从模型层到编译层的最小证据闭环，不出现“图能看见，但没有办法证明”的断点。

---

## 十一、首版边界

为了避免再做成一个“大而散的图工具”，首版需要明确收边界。

### 11.1 首版要做

- 一个统一工作台
- 一套数据旅程缩放体验
- 一组内置 case
- 一套对比能力
- 一套证据面板

### 11.2 首版不做

- 任意文件自由拼装分析
- 全量整网 tracing
- 自动异常诊断建议
- 多人协作、批注和分享
- 在线调试和运行控制

---

## 十二、后续路线

如果首版验证成立，后续再扩三件事：

- 从内置 case 走向用户自带 case
- 从单次对比走向时间序列和 timeline
- 从证据展示走向半自动诊断建议

PTO 的长期方向不是“再加更多图”，而是逐步成为一个面向数据旅程的调试与分析工作台。
