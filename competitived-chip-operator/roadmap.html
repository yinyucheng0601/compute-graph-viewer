P0 · Task Success

先解决三类最低谷：装不对、错不明、慢不知改哪

优先级模块覆盖触点为什么现在做首版验收
P0Stack Doctorversion matrix、container、framework ABI、arch support环境失败发生在旅程最早期，直接阻断试用；ROCm 的兼容矩阵说明这不是文档问题，而是产品入口问题。本机扫描后给出 supported / drift / rebuild / unsupported，并生成修复命令。
P0Kernel Error Explainer错误定位、源码行、坐标复现、reference diff调试低谷最深；CUDA Compute Sanitizer 和 Triton interpreter 分别证明源码定位和前置解释的价值。非法访问报告包含 kernel、源码行、program/thread 坐标、疑似原因、下一条命令。
P0Profile Experiment Notebook指标入口、source correlation、baseline、rule 建议性能是 GPU 产品核心价值；Nsight 的 baseline/rule/source correlation 是成熟标杆。每条建议包含证据指标、疑似原因、最小改动、预期指标变化、实验结果。

依赖关系

Stack Doctor 是所有模块前置；Error Explainer 依赖源码映射；Profile Notebook 依赖 baseline 存储。

首版边界

不做完整 IDE；先覆盖 CLI/HTML 报告、JSON 输出和可复制命令。

核心指标

首次可运行时间、非法访问定位时间、每次 profile 是否生成可复现实验记录。

竞品差距

CUDA 工具深但重；ROCm 数据开放但解释弱；Triton 快但底层链路不透明。

P1 · Productivity

再提升开发效率：从 reference 到 benchmark 的一体化算子工作台

优先级模块覆盖触点设计参照首版验收
P1Operator Labskeleton、shape/stride、autotune、reference/testTriton 的 Python-first、@triton.jit、autotune 和 benchmark 工作流。选择 op 类型后生成 kernel、reference、test、benchmark、export 模板。
P1Compile Explanation View编译日志、IR 可视化、target arch、自动优化解释CUDA PTX/SASS、ROCm LLVM、Triton MLIR 都说明编译层是性能解释链核心。源码行可跳到 IR/指令/寄存器风险，编译失败给最小复现命令。

P1 设计原则

01
模板不是脚手架，而是工作流必须包含 reference、测试、benchmark、profile、export。
02
autotune 必须可解释显示候选配置、失败原因、资源占用和选择理由。
