01 · Runtime Tiling

Tiling 不是写死数字，而是运行时函数

真实 CANN 算子里，tiling 函数签名通常是 ge::graphStatus TilingFunc(gert::TilingContext* context)。每次算子被调用，函数读取真实 shape、属性、dtype 和平台信息，然后写回 tiling data 与 tiling key。

编译期：TilingParse

保存与 shape 无关的硬件信息，例如核数、SoC 版本、UB/L1 大小。它给运行期决策提供环境边界。

CompileInfoSoCbuffer size

运行期：TilingFunc

通过 context 读取当前输入 shape，再计算 block、tile、split、workspace、key 等具体值。

GetInputShapeSetTilingKey

Kernel：模板实例

tiling key 把动态 shape 映射到不同 kernel 模板实例。每个实例可能有不同 unroll、buffer 和分支策略。

TILING_KEY_ISOpTemplate
两阶段 tiling 模型
这张图强调分工：硬件边界在编译期缓存，shape 决策在运行期发生，kernel 通过 tiling key 进入不同模板路径。

02 · Shape Contract

开发者不决定输入 shape，只决定如何覆盖 shape

输入 tensor 的 shape 来自调用方。LLM 推理算子常把除最后一维外的所有维相乘当 row/token 数，把最后一维或 weight/gamma 维当 feature/hidden。落到 MatMul，最不稳定的是 M=batch×seq。

算子 / 场景shape 读法tiling 含义
rms_normnumRow = ∏(前导维)，numCol = ∏(gamma 维)row 是 token 展平后的行数，col 是归一化 feature 宽度。
swiglubs_ = ∏(前导维)，d_ = 最后一维，splitD_ = d_/2B×S 变成动态 token 数，hidden 被拆成两个门控分支。
MatMul x @ WM=batch×seq，K=hidden，N=输出维K/N 来自 weight，通常固定；M 随请求长度和 batch 变化。

Prefill：M 大

一次处理整个 prompt，seq 较大，MatMul 更接近饱和计算场景。base tile 可以切得更大，算力更容易跑满。

promptlarge Mfuller compute

Decode：M 可能小到 1

逐 token 生成，seq=1，batch 也可能很小。MatMul 退化成瘦长形状，核利用率、尾块和 memory traffic 变成主要矛盾。

seq=1M=1low intensity
关键纠正：开发者脑中更接近“我的 x 是 [batch, seq, 4096]”，不是先验固定的 “M=512”。工具输入也应从场景范围出发，而不是让用户直接盲调 tile 参数。
03 · Dynamic Shape & Tiling Key

同一个 binary 靠 tiling key 覆盖一片 shape

动态 shape 不是一个 kernel 模板硬扛所有情况。运行期会按真实 shape、dtype、可选输入和属性计算 key，再进入 kernel 里的不同模板实例。每个 key 本质上是一段策略区间。

04 · Hardware Envelope

第一层问题是：base tile 能不能塞进片上 buffer

硬件容量是 tiling 的硬边界。超过 L0A、L0B、L0C、L1 或 UB 的 base tile 不是慢一点，而是不合法。double buffer 不是改变合法容量，而是在合法基础上要求再留一份空间。

芯片L0AL0BL0CL1UBbtSize
Ascend 950 / DAV_351064KB64KB128KB512KB118KB0
Ascend 910B / 310B / DAV_220164KB64KB128KB512KB约 192KB1024B
Ascend 311332KB32KB64KB512KB118KB0
L0AbaseM * baseK * dtypeBytes ≤ L0AA tile 必须完整放入 L0A。
L0BbaseK * baseN * dtypeBytes ≤ L0BB tile 必须完整放入 L0B。
L0CbaseM * baseN * 4 ≤ L0CL0C 按 FP32 4 字节累计结果算。
合法性包络与提速空间
三个反直觉点：L0C 按 FP32 4B 算；double buffer 要两份空间；baseM/baseN 需要按 C0=16 对齐，否则尾块和核利用率会变差。
05 · Cost Model

塞得下只是合法，性能还要看算力强度和访存

调研笔记里确认源码内部有启发式 cost model：算力强度、访存估算、cycle 估算、double buffer 开关和 iterator order 都会影响选择。但这些反馈没有作为 per-shape 曲线直接呈现给开发者。

compute intensity

computeCycle / memoryTraffic 越高越好，是最接近性能甜点的主指标。

memory traffic

源码估算 A、B 访存项；调研笔记特别记录没有 C 项。

block cycles

CalculateBlockCycles(baseM, baseN, baseK) 估算计算耗时。

DB decision

比较 dbOnPipeTime 与 dbOffPipeTime，选更短路径。

开发者缺失的是 per-shape 决策曲线
产品机会：现有代码能算，但没有把“为什么这个 shape 选这个 tile / key”解释出来。可视化工具应当沿 M 扫描，把合法区、策略翻转点和性能趋势放到同一张图里。
06 · Chip Path Difference

结构性芯片差异主要出现在融合算子的数据通路

910B 与 950 不只是 buffer 大小差异。关键 feature flag 是 ifSupportL0CToUB：950 支持 L0C 到 UB 的直接通路，910B 不支持。这对纯 matmul 影响很小，但会强烈影响 matmul + vector epilogue 的融合路径。

芯片ifSupportL0CToUBFixpipe融合路径含义
910B / DAV_2201falseV220Cube 结果需先落 GM workspace，再由 Vector 从 GM 读入 UB。
950 / DAV_3510trueV310Cube 可从 L0C 直连 UB，Vector epilogue 不必绕 GM workspace。
Fusion 路径：910B 绕 GM，950 直连 UB
07 · Safe Default

开发者的起点通常是“合法默认”，不是从零猜 tile

MatMul/Cube 族有内置自动 tiling 算法；自定义或 vector 族通常复制结构相近的现成算子，再改 shape 映射和分支。这个默认能让算子先跑起来，但不保证所有 shape 都快。

MatMul / Cube：自动 tiling

MatmulApiTiling::GetTiling(TCubeTiling&) 会根据 shape 和硬件进入算法流程，算出合法的 TCubeTiling。调研结论是它没有专门 dynamic 路径，也不缓存，每次从头算。

safelegal firstnot always fast

每个算子一个独立 tiling.cpp，骨架通常一致：读 shape → 算 factor → 设 key。开发者主要改 shape 映射、分支阈值和边界处理。

08 · Breakpoint Map

动态 shape 验证的关键是自动找断点

外边界通常由 serving 配置给定：M ∈ [1, max_batch × max_seq]。真正危险的是范围内部不平滑：对齐边缘、尾块、tiling key 阈值、full-load 到 split 的翻转，以及最大 shape 的 workspace/fit 风险。

断点为什么危险能否解析定位
M=1decode 最瘦场景，算力强度低，往往是最慢点。可以，固定下界。
对齐边缘非 16 或非 base 倍数触发尾块和核利用率下降。可以，整除关系。
尾块shape % tile ≠ 0，末核或末轮工作量少。可以，余数关系。
tiling key 阈值开发者分支条件造成策略断崖。可以，读取分支条件。
full-load → split超过 L1 整块装下临界后策略突变。可以，容量不等式。
最大 shapefit、workspace、blockDim 上限风险。可以，边界代入。
沿 M 扫描：从手猜样例到危险地图
验证策略：正确性用 golden 对比，性能用 profiler；工具先帮开发者选中最该测的 shape，把“猜测试矩阵”变成“覆盖断点地图”。
09 · Product Position

工具定位应从“切片演示器”升级为“动态 shape 决策台”

Part I 的 trace 回放回答“某一步在碰哪一片”；这篇真实开发调研补上“tiling 该怎么定、为什么难”。二者合在一起，产品主线应是：输入场景范围，载入安全默认，扫描断点和约束，解释每个 shape 的策略选择。

短期页面能力

01
场景输入芯片、batch/seq 范围、K/N、dtype、是否 fusion，不把 tile 参数作为第一入口。
02
合法性包络显示每个候选 base tile 是否通过 L0A/L0B/L0C/L1/UB/对齐约束。
03
断点扫描沿 M 自动标出 decode、尾块、key 阈值、full-load 翻转和最大边界。

后续增强

04
性能趋势叠加 compute intensity、memory traffic、DB 开关和 profiler 实测点。
05
Key 策略解释把源码分支条件映射成 shape 区间，让“为什么换 key”可见。
06
Trace 联动对某个断点 shape 执行 trace 回放，继续回答“当前 step 碰哪一片”。
输入不是调参开发者给的是部署场景和 shape 范围。

Terms · 术语表

阅读这份白皮书需要区分的概念

术语含义在产品里的展示建议
TilingFunc运行时读取 context 并计算 tiling 的函数。显示输入 shape、硬件信息、输出 tiling data 和 key。
TilingParse编译期解析并保存与 shape 无关的信息。放在“环境/平台”侧栏，不当成用户调参。
Tiling Key运行期策略分流编号，kernel 根据它进入不同模板实例。映射为 shape 区间和分支条件。
M / K / NMatMul 维度；推理里 M=batch×seq 最动态，K/N 多来自 weight。把 batch/seq 范围转换成 M 轴。
Base Tile单个基础计算块，必须满足片上 buffer 容量和对齐约束。在约束图中显示 fit / fail 原因。
Double Buffer用两份 buffer 重叠搬运和计算的提速策略。单独作为第二层优化，不混入第一层合法性判断。
Full-load / Split数据能否整体装入某级缓存导致的策略翻转。作为 M 轴上的重要断点。
GM Workspace融合路径里中间结果落到 GM 的临时空间。在 910B fusion 路径中突出带宽代价。
