# 业务理解后续研究主题判断

这篇笔记不是在重复解释 CANN、DeepSeek 或 PyPTO 的基础概念，而是基于当前业务理解文件夹里已经完成的几篇笔记，主动判断下一阶段最值得投入研究的主题。

先说结论：**下一阶段不应该继续停留在“概念扫盲”层，而应该转向“把已有理解变成可落到工具里的结构化能力”。**

目前这几篇文档，已经把三条关键链路分别铺开了。

[`developer_doc_zh.md`](/Users/yin/pto/业务理解/developer_doc_zh.md) 解释了 Python 代码怎样一步步变成初始 IR，回答了 `Source / Python AST / Doc AST / Liveness / Initial IR / Lazy Execution` 这条前端链路。  
[`cann_adaptation_guide.md`](/Users/yin/pto/业务理解/cann_adaptation_guide.md) 解释了 DeepSeek 适配 910B 时，芯片、算子、精度、动态 shape、KV cache、通信和性能调优这些层是怎么叠在一起的。  
[`deepseek_910B_tiling_guide.md`](/Users/yin/pto/业务理解/deepseek_910B_tiling_guide.md) 则更进一步，把“tiling 在 910B 上是如何一步步定出来的”讲到了子图切分、L1 reuse 和 profile 的层面。  
而 [`PROJECT_INDEX.md`](/Users/yin/pto/业务理解/PROJECT_INDEX.md) 给出了产品层的路线图，指出当前工具在 `精度调试 / 性能调优 / 架构洞察` 三条线上的已完成能力与明显缺口。

把这四篇放在一起看，下一阶段最值得做的，不是再写一篇“什么是 BF16”或者“什么是 matmul”，而是研究：**这些理解怎样真正穿到 graph 工具的页面、数据协议和交互模型里。**

## 一、优先级最高的主题：建立统一的“阶段模型”

我认为接下来最该研究的主题，不是某个单点技术，而是：

**同一个 DeepSeek kernel，如何在工具里统一展示 Source、Initial IR、Pass IR 和 Swimlane 四个阶段。**

这是当前所有笔记里最明显的结构性缺口。

为什么这么说？  
`developer_doc_zh.md` 已经把 `Source -> Python AST -> Initial IR` 讲清楚了。  
`PROJECT_INDEX.md` 又明确写了产品目标里有“代码到执行的链路打通”。  
但现在工具的三块能力其实还是分裂的：

- `pass-ir/index.html` 偏 Pass IR
- `source-flow/index.html` 偏源码 + 控制流
- 已有基础泳道图查看，但还缺跨阶段联动与统一阶段模型

也就是说，**业务理解已经有了“全链路”的文字说明，但产品还没有一套统一的阶段视图模型。**

这个主题要回答的核心问题应该是：

1. 哪些阶段应该用树，哪些阶段应该用 DAG，哪些阶段应该用泳道图？
2. 不同阶段之间，节点的身份如何对齐？靠 `semantic_label`、源码行号、tensor 名称，还是更稳定的跨阶段 ID？
3. 用户在一个阶段点到的对象，如何在其他阶段找到“同一个东西”？
4. 哪些信息是每个阶段都该保留的，哪些只属于某一阶段？

如果这个主题研究透了，后面很多功能都会自然落地，比如：

- `Source -> Initial IR` 的点击联动
- `Initial IR -> Pass IR` 的节点演化
- `Pass IR -> Swimlane` 的性能解释

从产品价值看，这会是最核心的一步。因为一旦这条链打通，工具就不再只是“看图”，而是变成“解释代码如何变成执行结果”的系统。

## 二、第二个最值得研究的主题：tiling 怎样影响子图切分和泳道图

前面的 tiling 笔记已经提出了一个非常关键的结论：**tile 不只是算子内部切块参数，它还会反过来影响子图分割和合并。**

这件事非常重要，因为它直接决定性能调优界面该怎么做。

如果这个结论只是留在文档里，开发者读完之后大概知道“tile 会影响 pass”；但如果产品要把它做成可视化能力，就必须继续研究下面这些问题：

1. 具体是哪些 pass 在利用 tile 信息？  
   是 `L1ReuseMerge`、`NBufferMerge`、`partition` 这类 pass 在起作用，还是还有其他更底层的图切分逻辑？

2. 哪些 tile 变化最容易触发子图形态变化？  
   是 vector tile 一致性更关键，还是 cube tile 规模更关键？

3. 工具里应该怎样解释“这两组 tile 为什么导致子图不一样”？  
   是做前后 diff，还是做“因果说明卡片”？

4. 当用户看到泳道图里任务变碎、出现气泡时，怎样把它追溯回“tile 设置不合适”这个原因？

这个主题之所以优先级很高，是因为它正好位于几篇文档的交叉处：

- `deepseek_910B_tiling_guide.md` 解释了 tile 和子图切分的关系
- `PROJECT_INDEX.md` 明确写了性能调优线还缺泳道图
- 当前 graph 工具其实已经有 Pass DAG 和源码视图，但还缺“tile -> 子图 -> 任务”的因果桥梁

换句话说，这个主题不是孤立研究，而是把当前业务理解直接变成功能设计的最短路径。

## 三、第三个主题：DeepSeek 在 910B 上的 profile 选择机制

如果前两个主题回答的是“怎么解释”，那么第三个主题回答的是“怎么运行”。

从目前的文档看，已经可以明确知道：DeepSeek 在 910B 上不会只用一套 tiling。`decode`、`prefill`、大 batch、小 batch、长 cache、短 cache，都可能命中不同 profile。

因此，下一个非常值得专门研究的主题是：

**运行时到底怎样在多套 profile 之间选择。**

这个主题不只是编译问题，也是产品问题。因为一旦你想在工具里做“推荐 profile”或者“比较不同 profile”，就必须先知道 profile 的选择依据是什么。

这个主题建议重点研究四类问题：

1. Profile 的分桶维度到底有哪些？  
   是 `batch`、`s1`、`s2`、`topk`、`block_size`、`cache length`，还是还要加上 `dtype`、`layout` 甚至量化策略？

2. 编译缓存和 profile 的关系是什么？  
   一组 tiling 是否对应一份独立编译结果？缓存命中粒度到什么程度？

3. 运行时如何决策？  
   是根据 shape 直接路由，还是先走粗粒度规则，再落到 profile 列表？

4. 产品上要不要把 profile 选择过程显式展示出来？  
   对开发者来说，看到“这次请求为什么命中了 profile A 而不是 B”通常是有价值的。

这个主题的成果，很可能不是一篇纯概念文档，而是一张“shape bucket -> profile -> 编译产物 -> 运行时命中”的关系图。它会直接支撑后续的推荐、对比和自动调优功能。

## 四、第四个主题：KV Cache 和动态 shape 的可视化模型

当前文档里，`KV cache` 和 `动态 shape` 已经在两篇笔记里都出现了：

- `developer_doc_zh.md` 从 `Lazy Execution` 的角度解释了动态维度是如何绑定真实输入 shape 的
- `cann_adaptation_guide.md` 从部署和推理的角度解释了 `KV cache` 为什么是大模型专属重点

但对产品来说，最难的地方不是“知道它们重要”，而是“知道该怎么画”。

所以我认为第四个很值得研究的主题是：

**如何在 graph 工具里展示动态 shape 和 KV cache，而不是只把它们写在属性面板里。**

这个主题为什么重要？因为大模型很多性能和正确性问题，根源都不在普通张量，而在：

- 本次调用到底绑定了什么 shape
- prefill 和 decode 进入了不同分支
- cache 是读旧值还是写新值
- cache index 是怎么决定写入位置的

如果工具里只能看到 `token_x: [t, h]`，却看不到 `t` 是怎样绑定的、`cache_index` 如何驱动 `kv_cache` 更新，那很多核心问题仍然解释不清。

这个主题建议往两个方向研究：

第一，语义模型。  
要先定义哪些对象需要被视为“动态对象”：

- `SymbolicScalar`
- `shape bucket`
- `cache slot`
- `cache update edge`

第二，视觉模型。  
要判断这些对象该怎么画才不会把主图淹没：

- 是在 DAG 里做 badge？
- 是单独做 runtime overlay？
- 还是用边标签、时间轴或 cache heatmap？

这是一个典型的“不是技术知识不够，而是产品表达还没设计好”的主题。

## 五、第五个主题：从单卡 kernel 调优，走向多卡拓扑视图

`cann_adaptation_guide.md` 已经把一个事实讲得很清楚：对于 DeepSeek 这类模型，适配 910B 不只是单卡问题，`tp=8`、`HCCS`、多机多卡通信都会影响最终部署方式。

但从当前工具路线看，这一层几乎还是空白的。

所以第五个适合继续研究的主题是：

**如何把“整机 8 卡 / 多机多卡”的并行拓扑，用图工具表达出来。**

这和当前的 `mvp` 架构图其实很搭。因为 `mvp` 已经有多层级结构浏览的能力，只是现在它更多表现的是模型结构，而不是部署拓扑。

如果沿着这个方向继续研究，建议重点回答：

1. 什么对象应该进入“部署拓扑图”？  
   是 NPU、HCCS、tensor parallel group、expert parallel group，还是更高层的服务节点？

2. 怎样把“模型结构图”和“部署拓扑图”接起来？  
   是某一层映射到某张卡，还是某个子图映射到某个并行组？

3. 多卡图是独立一张图，还是应该作为架构图上的一种 overlay？

这个主题的产品价值非常明确：它会让工具从“算子开发调试工具”进一步长到“部署架构理解工具”。

## 六、第六个主题：AI 辅助 tiling 的产品形态

这个主题不是最紧急，但它很值得提前研究，因为它很可能会变成后面的差异化方向。

前面的 tiling 笔记已经把一个边界讲清楚了：很难指望用一个闭式公式，一次性解出 DeepSeek 全部 kernel、全部场景、连带子图切分和调度一起的全局最优解。但这并不意味着 AI 做不了 tiling。

真正值得研究的问题是：

**如果把 AI 辅助 tiling 做成产品，它应该输入什么、输出什么、以什么方式和开发者协作。**

这个主题适合从产品视角而不是算法视角切入。需要先回答：

1. AI 的输入是什么？  
   是当前图结构、shape、平台规格、已有 profile、泳道图统计，还是这些全部？

2. AI 的输出是什么？  
   是一组 tile 建议、一条解释、一份 profile diff，还是“建议先调 vector tile，再调 reuse”这样的步骤建议？

3. 人机交互方式是什么？  
   是一键自动调优，还是让开发者在若干候选 profile 中选？

4. 怎样让建议可解释？  
   对这类工具来说，“为什么这么建议”比“建议本身”更重要。

这个主题的价值在于，它不会只停留在“AI 能不能做”，而是把问题推进到“如果要做成产品，最小可行版本是什么”。

## 七、反而不建议优先投入的主题

除了“该做什么”，也需要明确“暂时不要先做什么”。

基于当前这些笔记，我反而不建议下一阶段优先投入下面几类研究：

第一，继续写更多泛化的 CANN 基础概念文档。  
现有笔记已经足够支撑产品经理和入门开发者建立基础认知。继续写“什么是 BF16”“什么是 KV cache”这类单点扫盲，边际收益会明显下降。

第二，在没有统一阶段模型之前，继续扩展新的孤立页面原型。  
如果 `Source / Initial IR / Pass / Swimlane` 之间的身份映射还没研究清楚，再做更多单页 demo，很容易让能力越来越散。

第三，过早研究 VSCode 插件化。  
`PROJECT_INDEX.md` 已经把它放在后面，这是合理的。当前最大问题不是入口形式，而是核心模型和解释能力还没打通。

## 八、我给出的建议顺序

如果要把接下来的研究主题排成一个实际可执行的顺序，我建议这样走：

**第一阶段：先打“链路解释”基础。**

先研究：

1. 统一阶段模型：Source / Initial IR / Pass / Swimlane
2. Tiling 对子图分割和泳道图的影响
3. Profile 选择机制

这三个主题一旦研究透，产品层就能第一次真正回答：

```text
这段代码为什么会变成这张图
这张图为什么会被切成这些子图
这次运行为什么会命中这套 profile
```

**第二阶段：再补“运行时对象”的表达。**

然后研究：

4. 动态 shape 和 KV cache 的可视化模型
5. 单卡 kernel 调优和多卡拓扑之间的连接方式

这一步做完，工具就能从“图解释器”往“部署理解器”走。

**第三阶段：最后研究智能化助手。**

等前两阶段都清楚了，再研究：

6. AI 辅助 tiling / profile 推荐

因为没有前面的结构化模型，AI 建议也很难做得可解释。

## 九、压缩成一句判断

如果只保留一句结论，我的判断是：

**接下来的研究重心，应该从“继续理解概念”转向“把 Source、IR、Tiling、Profile、Runtime 串成一条可视化解释链”，这会比继续补充零散技术知识更有产品价值。**
