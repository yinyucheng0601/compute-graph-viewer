界面示意 05：同一个算子的三种工作流

CUDA / HIP fused op workflow
kernel skeleton
__global__ void fused_op(...) { int tid = threadIdx.x; int row = blockIdx.x; // explicit memory, sync, reduction }

优势是控制细；成本是线程、同步、内存和编译参数都要显式管理。

Triton fused op workflow
program skeleton
@triton.jit def fused_op(x, y, BLOCK): pid = tl.program_id(0) offs = pid * BLOCK + tl.arange(0, BLOCK) vals = tl.load(x + offs, mask=...) tl.store(y + offs, vals)

优势是 tile 级表达更紧凑；风险是编译器决策和后端差异要解释清楚。

界面示意 06：编译解释层

编译器栈需要从隐藏后台变成可追问的解释层：源码、IR、指令、寄存器、occupancy 和 profiler 指标要能互相跳转。

CUDA compile chain
nvcc
.cu → nvcc → PTX → ptxas → cubin / SASS

价值：专家能从 PTX/SASS 和 profiler 指标回到源码策略。

ROCm compile chain
hipcc
.hip → hipcc → clang → LLVM AMDGPU → code object

价值：开放 LLVM 链路适合接入自研 pass 和解释器。

Triton compile chain
jit
Python AST → Triton IR → MLIR dialects → LLVM/backend

价值：高层 DSL 的自动优化需要默认解释“为什么这样生成”。
