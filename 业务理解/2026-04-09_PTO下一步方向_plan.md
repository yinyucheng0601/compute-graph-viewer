# PTO 下一步方向 Plan

更新时间：2026-04-09

本计划基于三类材料综合判断：

- `pto/业务理解` 中已有业务理解笔记、PRD 和研究记录
- `pto/` 中现有模块能力与代码结构
- `gitcode/pypto-master`、`gitcode/pto-isa-master` 中可读取的官方文档与工具说明

本文将用户提到的“优化 coden 上板执行”解释为：**优化 codegen 到板端执行的可解释性、调优闭环和观测能力**。

## 一句话结论

PTO 下一步不该再继续做零散页面，而应该收敛成一条主线：

**以数据流为第一组织原则，打通 Source / Function / ControlFlow / Pass IR / Execute Graph / Swimlane / Memory 的多粒度调试工作台。**

## 优先级排序

1. 统一阶段模型与单工作台入口
2. ControlFlow 与 Pass / Swimlane 的跨阶段联动
3. Op 参数嵌套与函数嵌套可解释性
4. 动态 shape / 动态 GM / KV cache 生命周期可视化
5. 上板执行链路与 profile 命中解释
6. 片上资源视图：寄存器 / Tile / Memory / Sync
7. 跨 IDE / 跨工作区接入
8. AI 辅助调优与可解释推荐

## 阶段计划

### Phase 1：先把主干打通（1-3 周）

目标：把 PTO 从“多入口原型集合”升级成“统一工作台骨架”。

交付：

- 新建统一入口页面，提供单对象旅程：
  `Code -> semantic_label -> Execute Graph -> Block Graph -> Swimlane`
- 抽象统一对象模型：
  `functionId / semanticLabel / callOpMagic / rootHash / leafHash / subgraphId / rawTensorMagic`
- 把现有 `pass-ir`、`swimlane`、`source-flow` 的联动桥接收敛成统一 session 数据协议
- 补齐 `program.json` 绑定状态、失败原因、缺失文件提示，不再静默失败

建议优先复用：

- `pass-ir/`
- `swimlane/`
- `source-flow/`
- `execution-overlay/`

### Phase 2：补控制流解释层（第 3-6 周）

目标：把“源码 loop / tile loop / 编译后 controlflow”明确分层，而不是混成一张流程图。

交付：

- 增加四层 loop 标记：
  模型层、源码层、tile 遍历层、编译后 controlflow 层
- 在 ControlFlow 面板中增加 path 与 step 解释：
  `PATH0_6/8/10...` 不只显示 magic，要显示对应的 unroll step
- 建立 `loop_unroll -> path -> callOpMagic -> swimlane task` 的追踪链
- 支持从某个 task 反查所属 loop 档位、首块/中块/尾块分支

### Phase 3：补结构解释层（第 4-8 周）

目标：让用户真正看懂“函数嵌套”和“op 参数嵌套”。

交付：

- Function Stack 视图：
  显示 `frontend.jit` 入口、`pypto.function`、`frontend.function`、ExpandFunction 前后边界
- Op Detail 2.0：
  把平铺参数表升级成嵌套参数树
- 参数 provenance：
  显示参数来自源码声明、动态维度、tile 设置、pass 注入，还是 codegen 派生
- Tensor / rawTensor / slot / offset / asis / tobe 联合视图

### Phase 4：补运行时对象层（第 6-10 周）

目标：让动态对象可视，而不只是停留在属性面板。

交付：

- Dynamic Shape Overlay：
  可视化 `SymbolicScalar` 的绑定值、shape bucket、命中 profile
- Dynamic GM / KV Cache 视图：
  展示分配、复用、释放、写回、命中
- Memory Timeline：
  把 `mem_viewer` 从静态回放升级为与 Execute Graph / Swimlane 联动
- 支持在 Block Graph 或 Swimlane 中高亮“当前存活 tensor”

### Phase 5：补硬件解释层（第 8-12 周）

目标：从“图可视化”走到“片上资源可解释”。

交付：

- Tile / Memory / Register 视图
- 指令级 overlay：
  `TLOAD / TSTORE / TASSIGN / TSYNC / TSETFMATRIX` 等关键原语的资源说明
- Manual / Auto 模式切换观察
- 子图层的 sync / wait / barrier 可视化

### Phase 6：补产品化接入层（并行推进）

目标：降低 VSCode 依赖，走向跨 IDE、跨工作区、跨环境。

交付：

- 文件夹导入模式
- 独立 Web 版工作台
- VSCode 插件保留但不再作为唯一主路径
- Remote / 容器 / 本地目录的统一适配协议

重点：

- `pypto.loop`

### Track B：Op 参数嵌套

重点：

- operation 参数树
- Tensor 与 rawTensor 关系
- offset / memory tier / subgraph 来源
- tile 与 pass 配置如何影响 op 形态

产物：

- 可折叠参数树
- 参数来源标记
- 参数 diff 视图

### Track C：函数嵌套

重点：

- `frontend.jit`
- `pypto.function`
- `frontend.function`
- ExpandFunction 前后图变化

产物：

- Function Stack
- Inline / Expand lineage
- “源码函数”与“执行函数”映射图

### Track D：动态 GM / Memory

重点：

- 动态 shape
- KV cache
- GM <-> Tile 搬运
- rawTensor 生命周期

产物：

- 生命周期时间线
- 分配 / 复用 / 释放事件
- cache update 热区图

### Track E：跨 IDE

重点：

- VSCode 当前强绑定点
- 工作区自动发现

产物：

- IDE 无关的数据装载层
- 本地 Web 独立版

### Track F：寄存器与片上资源

重点：

- FMATRIX

产物：

- 片上资源面板
- 指令契约卡片
- Auto vs Manual 对照

### Track G：上板执行 / Codegen

重点：

- Tensor Graph -> Tile Graph -> Block Graph -> Execute Graph
- subgraph 切分 / 合并
- profile 命中
- root / leaf / callOpMagic

产物：

- Codegen 阶段 diff
- profile 命中解释
- tile -> subgraph -> swimlane 因果链

## 具体建议的产品形态

### 1. 主产品

`PTO Dataflow Workbench`

作为统一入口，整合：

- 代码视图
- Function / ControlFlow 视图
- Pass IR / Execute Graph / Block Graph
- Swimlane
- Memory / Cache

### 2. 辅助产品

`PTO ISA Inspector`

聚焦：

- Tile / GlobalTensor
- 指令契约
- 寄存器 / buffer / sync
- Auto / Manual 差异

### 3. 调优产品

`PTO Profile Diff`

聚焦：

- tile / pass 选项 diff
- subgraph 数变化
- 任务数、气泡、关键 task 时长变化

## 30 / 60 / 90 天里程碑

### 30 天

- 统一入口工作台上线
- `program.json` 绑定与跨视图定位稳定
- swimlane -> execute -> block 三跳稳定
- loop/path 基本解释能力上线

### 60 天

- Function Stack 与 Op 参数树上线
- dynamic shape / profile 命中解释上线
- `mem_viewer` 与主工作台联动
- Profile Diff 第一版上线

### 90 天

- GM / cache 生命周期可视化上线
- 片上资源视图上线
- 跨 IDE 独立运行版上线
- AI 调优建议第一版上线

## 暂不建议优先做的事

- 继续扩更多孤立 demo 页面
- 过早做重型 VSCode 插件化重构
- 继续写泛化基础概念扫盲，而不落到数据协议和页面能力
- 只做“更漂亮的图”，不补可解释性与跨阶段对象模型

## 最终判断

下一步最值得投入的，不是新的单页原型，而是三件事：

1. 统一对象模型
2. 统一阶段旅程
3. 统一调优闭环

只要这三件事成立，控制流、参数嵌套、函数嵌套、动态 GM、寄存器、上板执行和可视化就不再是七八条散线，而会收敛成一条产品主线。
