Part 1：认知层面，950 架构有什么更新
新手最容易犯的错：把 950 当成 910B 的算力增强版。
950 的关键变化在数据通路、低比特、RegBase、SIMT/SIMD、C-V 直连和通信硬化资源。写算子前，要先知道数据到底停在哪一层、被哪类执行单元消费。
AI Core 子系统升级

学习 950 时先把 AIC、AIV、SIU 分开：AIC 负责 Cube 主计算，AIV 负责 Vector/SIMD/SIMT，SIU 和片上互联影响多单元协同。调优时不要只问“哪个核忙”，要问“数据从哪里来、到哪里去”。

AIC AIV SIU CCU
数据通路升级

950 学习重点是减少 GM 往返，提高片上复用。最需要记住的路径是 GM、L2、L1、L0A/B/C、UB、Vector Reg File、SIMT Reg File 之间的关系。

低比特成为主路径

低比特不是把 dtype 改小。MXFP8/MXFP4 需要关心 scale 张量、scale 缓存、layout、LoadData 类接口、量化/反量化融合和精度验证。

HiF8 FP8 MXFP8 MXFP4
能力开发者要建立的判断优先联动的硬件对象
RegBase / SIMD连续规则表达式、规整向量融合，优先减少 UB 中间结果反复读写。GM - UB - Vector Reg File
SIMT离散索引、Gather/Scatter、Hash/Atomic、复杂分支，优先按 Thread Block / Warp / Lane 建模。GM - UB - SIMT Reg File；GM - SIMT Reg File
Cube 低比特MatMul、GEMM、qbmm、MoE 主计算要同时设计 scale、layout 和 L0A/B 组织。GM - L0A/B - Cube - L0C
C-V 融合MatMul 后处理、FIA、MatMul+Add、RmsNormQuant 等场景优先检查是否能少走 GM。L0C - UB；UB - L1；SSBuf

低比特数据类型速览

类型950 上的关注点写算子时先检查什么
HiF8 / FP8进入 MatMul、量化、反量化和端到端精度验证路径。确认输入输出 dtype、round / saturate 策略、与上游模型导出格式是否一致。
MXFP8主数据变窄后，scale 张量的搬运和复用会直接影响收益。把 scale 的 GM、UB、L1 生命周期写进 Tiling，而不是只改 LoadData 类型。
MXFP4压缩收益更明显，但 scale / layout / 伪量化误差也更敏感。先做 Golden 对齐和极值样本验证，再决定 scale 预取粒度与 tile group。
混合类型Cube 主计算、Vector 后处理和 Store 阶段可能使用不同精度。明确 cast、anti-quant、quant、activation 是否能融合，避免多次 GM 往返。

片上存储层次速查

层级开发者心智容易踩坑
GM / L2外层数据入口和跨核共享视角；不是所有中间结果都应该回到这里。把 workspace 当默认中转，导致 C-V 融合收益消失。
L1 / L0A / L0BCube 主计算前的数据组织，低比特 layout 和 Bank 访问会影响吞吐。沿用 910 Tiling 时忽略 L1 bank conflict、尾块和低比特 scale 对齐。
L0CCube 结果停留点；融合场景优先检查能否直接给 AIV。仍按 L0C - GM - UB 写旧路径，增加一次大块 GM 往返。
UB / Vector Reg FileAIV 侧 staging 与 SIMD Reg 计算路径；连续规则表达式尽量减少 UB 反复读写。RegBase 临时变量过多导致 spill，把寄存器优化变成 stall 来源。
SIMT Reg File离散索引、分支、Gather/Scatter 和线程式表达的寄存器路径。把离散数据硬套 SIMD，导致访存合并差、同步和边界问题难排查。

代际差异速查

主题910 / A3 代码里常见的隐含假设950 上要补的判断
Vector多按 Memory-based 路径理解，核心是 GM、UB、LocalTensor 和流水隐藏。连续规则数据看 SIMD Reg / RegBase；离散索引数据看 SIMT Reg File。
Cube + VectorCube 主循环和 Vector 后处理容易拆成多个算子，靠 GM 或 workspace 过渡。先检查 L0C2UB、UB2L1、DualDest、SSBuf，判断能否减少 GM 往返。
低比特常被当成 dtype 替换，scale、layout、搬运接口放到后面考虑。把 HiF8、FP8、MXFP8、MXFP4 当作主路径能力，提前设计 scale cache。
线程模型离散访存、分支和线程模型不是大多数 Ascend C 新手的第一关注点。SIMT 场景按 Thread Block / Warp / Lane 设计，并把 sanitizer 放进调试预设。

Vector 算子：连续规则数据走 SIMD Reg

适合场景

Add、RMSNorm、Quant、激活函数、规则表达式融合。950 上的收益来自 RegBase / VF 融合减少 UB 中间结果往返，但寄存器变量过多会引发 spill。

RegTensor MaskReg AddrReg SIMD VF
硬件路径

主路径是 GM - UB - Vector Reg File，结果再从 Reg 写回 UB / GM。若数据离散，优先不要硬套 SIMD。

// 伪代码骨架：规则向量表达式，先 GM->UB，再 UB->Vector Reg File
__global__ __aicore__ void VectorAdd950(GM_ADDR x, GM_ADDR y, GM_ADDR out) {
  // 1. Tiling 决定每个 block 的 GM offset 和 UB tile 大小
  CopyIn(x, y);                  // GM -> UB
  LoadToVectorReg();             // UB -> Vector Reg File
  SimdVfAddOrFuse();             // SIMD VF / RegBase 计算
  StoreFromVectorReg();          // Vector Reg File -> UB
  CopyOut(out);                  // UB -> GM
}

Cube 算子：低比特进入主计算路径

适合场景

MatMul、GEMM、GMM、qbmm、MoE。950 不只看 Cube busy，还要把低比特 scale、layout、伪量化、尾块和 L2/L1 复用一起纳入 Tiling。

L0A L0B L0C Scale Cache
硬件路径

主路径看 GM - L0A/B 到 Cube 主计算，结果进入 L0C；如果后面接 Vector epilogue，要继续检查 L0C 到 UB 的直连。

// 伪代码骨架：低比特 MatMul 先把数据与 scale 的生命周期一起设计
__global__ __aicore__ void LowBitMatmul950(GM_ADDR a, GM_ADDR b, GM_ADDR scale, GM_ADDR c) {
  LoadScaleToUB(scale);          // scale 不要每个 inner loop 反复从 GM 取
  LoadAtoL0A(a);                 // GM -> L0A
  LoadBtoL0B(b);                 // GM -> L0B
  MmadLowBit();                  // Cube 计算，结果在 L0C
  FixPipeOrEpilogue(c);          // 视场景写 GM 或接 C-V 直连
}

融合算子：Cube 结果尽量少走 GM

适合场景

MatMul+Add、MatMul+Activation、RmsNormQuant、FIA、Attention 后处理。先判断 Cube 结果能否通过 L0C2UB、DualDest 或 UB2L1 给 Vector 侧使用。

__mix__(aic_aiv) L0C2UB UB2L1 DualDest
硬件路径

关注 L0C - UB 和 UB - L1 的双向复用，以及 SSBuf / 指令序列如何让 AIC 与 AIV 协作。

Vector 算子补充：SIMT 处理离散数据

适合场景

索引、分支、Gather/Scatter、Hash/Atomic、稀疏或不规则访存。SIMT 是单指令多线程模型，Warp 内有 Lane，调试时要额外关注同步、越界和 GPR 压力。

Thread Block Warp Lane SIMT VF
硬件路径

对离散数据，要表达 GM - UB - SIMT Reg File 与 GM - SIMT Reg File 两类路径，避免把 SIMT 密集算子误写成纯 SIMD。

Part 2.2：如何把 910 算子迁移到 950
迁移不是改 CMake 版本号后重编译。
先扫描旧算子里的硬件假设：数据是否绕 GM、中间结果是否用 workspace、Vector 是否反复读写 UB、是否把低比特当普通 dtype、是否存在跨代不兼容 API。
旧写法 / 旧假设950 迁移检查建议动作
L0C 结果先写 GM，AIV 再从 GM 读是否属于 MatMul 后处理、FIA、RmsNormQuant 等融合场景优先评估 L0C2UB、DualDest、UB2L1，减少 workspace
Vector 中间结果全部在 UB / LocalTensor 之间流转是否是规则表达式或可融合的连续向量计算评估 RegBase / SIMD VF，减少 UB 往返
离散索引逻辑写成 SIMD 循环是否存在 Gather/Scatter、Hash/Atomic、分支发散评估 SIMT，按 Thread Block / Warp 建模
低比特只替换 dtypescale 是否从 GM 反复读取，layout 是否适配 MatMul把 scale 缓存、cast、round、saturate 和精度校验纳入设计
旧 Tiling 隐含 910 容量与核数L1 / L0 / UB / Reg / L2 命中假设是否仍成立重新做 Tiling baseline，再逐步启用 950 新通路
迁移重点 1：旧 GM 中转

看到 L0C -> GM -> UB，先不要急着照搬。950 的学习目标是判断能否让 Cube 结果更近地交给 Vector 侧。

迁移重点 2：Vector RegBase

Memory-based Vector 迁移时，先找表达式融合和临时变量；再判断哪些中间结果值得进 Vector Reg File。

迁移重点 3：低比特 scale

MXFP8/MXFP4 的 scale 生命周期要和主计算一起设计；否则低比特节省的带宽会被 scale 搬运吃掉。

坑点 01 / 融合路径
仍按 L0C→GM→UB 旧路径实现融合
910/A3 代码常把 AIC 输出写到 GM 或 workspace，再由 AIV 读回。950 上 MatMul 后处理、RmsNormQuant、FIA 等融合场景要优先评估 L0C→UB、DualDest、UB2L1 和 SSBuf。
对应动作：先画 C-V 数据流，再判断 Cube 输出布局和 DualDest M/N 对齐约束。
代码模板
// 910 / A3 常见路径：中间结果绕 GM 或 workspace
AIC: Mmad(a_l0a, b_l0b, c_l0c);
AIC: Copy(c_l0c, workspace_gm);      // L0C -> GM
AIV: Copy(workspace_gm, x_ub);       // GM -> UB
AIV: VectorEpilogue(x_ub, out_gm);

// 950 迁移方向：优先评估 C-V 直连
AIC: Mmad(a_l0a, b_l0b, c_l0c);
AIC: CopyL0CToUB(c_l0c, x_ub);       // L0C -> UB，或使用 DualDest
AIV: VectorEpilogue(x_ub, out_gm);   // UB -> Vector Reg / GM
坑点 02 / 低比特
MXFP4 / MXFP8 scale 没有预存到 UB
低比特节省的是主数据带宽，但 scale 如果在 inner loop 中反复从 GM 读取，会把收益抵消掉。矩阵步长还要同步检查 L1 bank conflict。
对应动作：把 scale 生命周期写进 Tiling；优先在 prologue 加载到 UB，并对 layout、padding、swizzle 做 review。
代码模板
// 风险写法：scale 随每个 tile 重复从 GM 读取
for (tile in k_tiles) {
  LoadA(tile);
  LoadB(tile);
  LoadScaleFromGM(tile);             // 容易成为额外 GM 搬运瓶颈
  MmadLowBitWithScale();
}

// 950 迁移方向：按 tile group 预加载并复用 scale
PreloadScaleToUB(scale_gm, scale_ub);
for (tile in k_tiles) {
  LoadA(tile);
  LoadB(tile);
  UseScaleFromUB(scale_ub);
  MmadLowBitWithScale();
}
坑点 03 / Vector 范式
SIMD / SIMT 范式选错
连续规则表达式适合 SIMD Reg；Gather/Scatter、Hash/Atomic、复杂分支更适合 SIMT。两者都属于 AIV/Vector 侧编程，但寄存器组织和调试方式不同。
对应动作：先按数据形态选范式，再决定是否用 MIX 组合 AIC/AIV。
判断模板
// 规则连续数据：SIMD / RegBase
if (access_pattern == contiguous && expression_is_regular) {
  CopyGMToUB();
  LoadUBToVectorReg();
  RunSimdVf();
  StoreVectorRegToUB();
}

// 离散索引数据：SIMT
if (access_pattern == gather_scatter || has_branch_divergence) {
  MapThreadBlockAndWarp();
  LoadOrAccessSimtRegFile();
  RunSimtVf();
  CheckSyncAndBounds();
}

跨代 API / layout 敏感点没有扫描
L1Buffer、L0A/B/C、DataCopy、LoadData、DumpTensor、低比特类型和 cube-only 模式都可能绑定旧硬件假设；Profiling 脚本也不能沿用旧 SoC 参数。
对应动作：迁移前先做关键字扫描，再逐项回到当前 CANN 头文件和工具链参数确认。

坑点 05 / 调优闭环
Pipeline stage、寄存器压力和尾块只凭经验调
Double Buffer、Pipeline stage、cross interval、GPR spill 和尾块均衡都会影响 bubble；只看总耗时很难定位是哪一级失效。
对应动作：用 Pipe 流水图、核间负载图、PC stall 和 register pressure 一起闭环。
检查模板
# 迁移 review 里固定这几个观测点
