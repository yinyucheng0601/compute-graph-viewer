Touchpoint 03 · Debugging

调试体验的目标是把 runtime fault 还原成可修改的源码原因

GPU 错误常常延迟暴露：非法访问、异步同步、数值偏差、shape/stride 错误、mask 漏洞。成熟工具要把“报错现象”变成“最短排查路径”。

01 错误定位

能否从 invalid read / assert / launch failure 回到 kernel、源码行、线程/程序 ID。

P0source line

02 断点变量

cuda-gdb/rocgdb 保留传统调试器能力，但学习和配置成本偏高。

专家工具

03 数值对齐

DNN 算子常需要 torch/reference 对比，定位误差从哪一层 tile 或 mask 开始扩散。

reference diff

Triton interpreter 先在 CPU/Python 侧收窄问题，是很强的体验启发。

优势工具链成熟，错误类型、源码关联、memory/race/initcheck 覆盖广。传统调试器心智可复用，适合底层工程师深入 kernel 状态。Python-first，能在进入 GPU 前定位 shape、mask、offset 逻辑问题。
劣势新用户不知道先跑哪个工具，异步错误仍需要同步点和最小复现。导览弱，工具名和底层组件多，迁移用户容易迷路。interpreter 有类型和间接访问限制，真实性能/并发问题仍需外部 profiler。
成功体验报错直接给源码行、线程坐标、访问地址和建议检查项。能把 wavefront/lane/LDS 状态暴露给专家，但需要更强上层解释。用户先用 reference diff 收窄到 Python 逻辑，再切 GPU 工具验证。
总结完整但入口多。能力在，导览弱。低门槛，但边界明确。

我们的方向：Kernel Error Explainer

核心理念：错误报告必须给“下一条命令”和“最可能原因”

01
统一错误分类把 launch failure、illegal memory、race、assert、numerical mismatch 映射到固定排查树。
02
源码关联 + 坐标复现报告 kernel 名、源码行、program/thread 坐标、shape/stride、mask 值，让用户能构造最小复现。
03
参考实现对比内置 torch/numpy reference diff，显示首次超差 tile、误差分布和可疑 load/store。
