# PTO UX 原型迁移与高保真维护工作流笔记

## 背景

当前 PTO 的 UX 交付链路大致是：

1. 交互设计师产出低保真或半成品原型，例如 `mem_viewer.html`
2. 维护者接手后，将原型拆分为模块化结构，例如：
   - `index.html`
   - `styles/main.css`
   - `js/*.js`
3. 再把视觉升级为 PTO 统一高保真风格，并并入 `launch`
4. 原型迁移过程中，一些细节交互、状态规则、信息层级可能丢失
5. 为了避免信息损耗，设计师额外提供 `*_doc.md` 作为设计说明与验收依据

这个工作流本质上不是“纯前端实现流程”，而是“设计资产翻译与保真迁移流程”。
核心风险不在代码，而在信息丢失、责任边界模糊、以及迁移后难以验证。

---

## 当前痛点

### 1. 原型是视觉稿，不是工程结构

设计师给出的 `mem_viewer.html` 往往更像一个一次性可运行原型：

- HTML / CSS / JS 耦合
- 状态逻辑写死
- 视觉 token 不一定符合 PTO 全局体系
- 与 `launch` 集成时缺少模块边界

这意味着迁移者不是“改样式”，而是在做一次半重构。

### 2. 迁移过程天然会损失交互细节

尤其容易丢的是：

- 悬浮逻辑
- 状态切换规则
- 动效节奏
- 信息显隐条件
- 面板之间的空间关系
- 异常态 / 空态 / 边界态

如果这些没有在 `doc` 中结构化表达，后续只能靠口头同步和截图回忆。

### 3. 视觉升级与功能迁移同时发生，容易混淆问题来源

例如：

- 是视觉升级导致布局变化
- 还是模块拆分导致交互失效
- 是 launch 集成导致样式被覆盖
- 还是设计说明本身不完整

如果没有中间层，排查成本会越来越高。

### 4. 验收标准不统一

设计师看的是“效果是否符合原型意图”；
工程维护者看的是“结构是否可维护、能否并入 PTO”。

如果没有统一检查表，就容易出现：

- 设计觉得“味道不对”
- 工程觉得“已经实现了”

---

## 推荐工作流

建议把流程拆成 4 个明确阶段，每个阶段产出不同资产。

### 阶段 1：原型输入标准化

设计师交付时，不要只给一个 `xxx.html`，而是固定交付三件东西：

1. `xxx.html`
   - 可运行原型
2. `xxx_doc.md`
   - 设计说明
3. `xxx_checklist.md`
   - 验收清单

其中：

#### `xxx_doc.md` 应包含

- 页面目标
- 主要模块
- 关键状态
- 用户操作路径
- 每个交互的预期反馈
- 哪些是必须保真，哪些可工程化调整

#### `xxx_checklist.md` 应包含

- 布局是否一致
- 信息层级是否一致
- 状态是否齐全
- hover / click / drag / collapse / empty 等是否保留
- launch 集成后是否仍成立

这样迁移者拿到的不是“代码 + 口头说明”，而是“代码 + 设计契约 + 验收契约”。

---

### 阶段 2：迁移前先做“结构映射”

在真正改代码前，先做一个简短的迁移映射文档，建议放在模块目录下，例如：

`mem_viewer/MIGRATION_NOTES.md`

内容只需要 3 部分：

#### 1. 模块映射

- 原型中的哪个区域，迁移后对应哪个文件 / 模块

例如：

- graph toolbar -> `styles/main.css` + `js/graph-viewer.js`
- op detail -> `js/op-detail.js`
- runtime status -> `js/playback.js`

#### 2. 保真项

- 哪些细节必须保留

例如：

- 节点点击后原地浮层
- 播放时当前 op 高亮
- mem buffer 使用 tier 色彩体系

#### 3. 可重构项

- 哪些允许为了 PTO 统一规范而调整

例如：

- 字体系统
- 阴影和圆角 token
- toolbar 布局方式
- launch 的入口形式

这一步可以极大降低“改着改着偏题”的概率。

---

### 阶段 3：代码层保持“双层结构”

建议每个 demo 模块都按两层维护：

#### A. 原型层

保留设计输入的参考资产，尽量不反复改：

- `mem_viewer.html`
- `mem_viewer_doc.md`

用途：

- 对照
- 回溯
- 讨论

#### B. 工程层

真正并入 PTO 的实现：

- `mem_viewer/index.html`
- `mem_viewer/styles/main.css`
- `mem_viewer/js/*.js`

用途：

- 高保真实现
- 可维护结构
- 与 `launch` 集成

关键原则：

- 原型层是“参考真相”
- 工程层是“产品真相”
- 两者不要混用

不要一边把 `mem_viewer.html` 当参考，一边又持续在里面做大量生产级修改。

---

### 阶段 4：迁移后做“差异验收”

每次迁移完成后，不要只看效果，建议固定做一次差异验收：

#### 视觉差异

- 模块层级是否一致
- 重点信息是否仍然突出
- 间距和对齐是否破坏原型节奏

#### 行为差异

- 点击 / hover / 拖拽 / 折叠 / 播放是否完整
- 默认态、选中态、进行态、完成态是否都有

#### 工程差异

- 是否已拆成独立模块
- 是否复用 PTO token / 组件 / launch 入口
- 是否把一次性写法替换为可维护实现

建议验收结果也沉淀为：

`mem_viewer/ACCEPTANCE_NOTES.md`

---

## 推荐的目录与资产规范

针对每个 demo，建议长期固定如下结构：

```text
mem_viewer/
  index.html
  styles/
    main.css
  js/
    playback.js
    graph-viewer.js
    op-detail.js
    ...
  data/
    ...
  MIGRATION_NOTES.md
  ACCEPTANCE_NOTES.md

mem_viewer.html
mem_viewer_doc.md
mem_viewer_checklist.md
```

其中：

- `mem_viewer.html` 是输入参考
- `mem_viewer_doc.md` 是设计说明
- `mem_viewer_checklist.md` 是验收标准
- `MIGRATION_NOTES.md` 是结构映射
- `ACCEPTANCE_NOTES.md` 是落地结果与已知偏差

---

## 角色分工建议

### 交互设计师负责

- 提供原型
- 提供说明文档
- 标注必须保真的交互和信息层级

### PTO 维护者负责

- 模块拆分
- 高保真风格统一
- launch 集成
- 技术约束下的合理重构

### 双方共同负责

- 对“哪些可以改，哪些不能改”形成共识
- 对迁移后的偏差做记录，而不是靠记忆

---

## 最值得建立的机制

如果只做一件最重要的事，建议是：

### 为每个 demo 建立“迁移契约”

即：

- 原型文件
- 设计说明
- 验收清单
- 迁移映射
- 落地偏差记录

这套机制的价值在于：

- 人员变动后仍可接续
- 一个 demo 改很多轮也不会失控
- 不会再依赖“这个细节我记得以前讨论过”

---

## 对当前 PTO 团队的具体建议

结合现状，最适合你的不是重流程工具，而是轻量规范：

1. 每个 demo 强制有 `doc + checklist`
2. 迁移前先写 `MIGRATION_NOTES`
3. 迁移后写 `ACCEPTANCE_NOTES`
4. 原型层与工程层分开，不混改
5. launch 集成只接受工程层模块，不直接挂原型文件

这会明显降低：

- 细节交互丢失
- 设计意图漂移
- 多人协作时的信息断层
- 后续维护者接手成本

---

## 一句话总结

PTO 当前的 demo 维护，本质上要从“接原型改页面”升级成“设计资产保真迁移系统”。

一旦把输入、迁移、验收三件事拆开管理，效率和一致性都会明显提升。
