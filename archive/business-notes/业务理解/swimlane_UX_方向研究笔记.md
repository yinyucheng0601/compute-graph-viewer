# Swimlane UX 方向研究笔记

> 写于 2026-03-24
> 背景：swimlane 工具已积累大量功能，但用户体验零散，需要重新找到把功能串联起来的叙事结构。

---

## 一、现状诊断

### 功能已经很全，体验却很碎

当前 swimlane 已实现：
- 完整的 trace parser（seqNo / rootHash / leafHash / callOpMagic / ioperand-hint / ooperand-hint 全部保留）
- Task 详情面板（Timing / Graph / Operand / Compare / Actions 五个区块）
- Before/After Diff 视图，含 delta 指标
- Lane 排序（按气泡 / 利用率 / 任务数 / 时长）
- 颜色模式切换（语义标签 / stitch / 子图）
- Stitch 序号过滤、区间选择、时间标记、搜索
- 底部分析卡（热点 lanes / 热点 labels / 诊断建议）
- program.json 绑定（跨视图跳转的数据桥梁）

功能密度已经很高。但用户打开页面后看到的是：一堆按钮、一堆面板、一堆数字，没有任何指引说明"你现在应该看什么"、"下一步去哪"。

### 根本问题

工具是**以功能为单位组织的**，不是**以用户在做什么为单位组织的**。

用户来这里不是为了"使用颜色模式切换"，他来这里是为了回答一个问题：
> "这次跑出来的 profile，到底哪里慢，为什么慢，我该怎么改？"

这个问题有自然的认知顺序，但工具现在把所有答案平铺在一个界面上，用户不知道从哪里开始。

---

## 二、核心洞察：Journal 作为叙事容器

### 把一次分析过程变成一篇 Journal

用户分析性能 profile 的过程，本质上是一次侦探推理：

```
看到现象 -> 提出假设 -> 找证据 -> 定位原因 -> 提出动作
```

每一步都可以对应 swimlane 里的一个交互动作：
- **看到现象** — 打开文件，看全局利用率、气泡分布
- **提出假设** — "是不是 AIC 有几条核特别空闲？"
- **找证据** — 点 lane、看 detail、用 measure 框选时间区间
- **定位原因** — 看 seqNo / rootHash / subgraph 归因，跳转 Execute Graph
- **提出动作** — Before/After Diff 确认优化效果

**Journal 的价值**：把这个过程记录下来，变成可复现、可分享、可迭代的分析文档。不只是截图加备注，而是每一个关键发现都带着当时的视图状态（缩放、高亮的 lane、选中的 task、框选的区间）。

## 三、用户旅程 x 功能对应关系

### 旅程 1：首次打开，建立全局认知

**用户问题**：这个 profile 整体情况怎么样？

**当前体验**：数字在 toolbar 里，底部有面板，但用户不知道该先看哪里。

**理想叙事**：
- 打开文件后，先呈现一张全局健康卡
  - AIC 利用率 / AIV 利用率 / 总 span / lane 数
  - 自动识别：是 cube 密集型 / vector 碎片型 / stitch 屏障型
  - 给出一句话诊断："AIV 利用率只有 28%，主要瓶颈在 vector 稀疏，建议从气泡最大的 3 条 AIV lane 入手"
- 这一步结束后，用户有了分析入口，而不是面对一堆空白泳道

---

### 旅程 2：找瓶颈，缩小范围

**用户问题**：哪几条 lane 最值得看？

**当前体验**：排序按钮有，但分散在 Analyze Card 里，用户需要主动去找。

**理想叙事**：
- Lane 列表视角：左侧 lane label 区域本身就是一个排行榜，在 lane 名称旁边常驻显示利用率或气泡占比的 minibar，不需要打开排序面板就能扫到差异：
  ```
  AIC_0  ████████████░░  85%
  AIC_7  ███░░░░░░░░░░░  23%  ← 视觉上就看出来了
  AIV_0  ▓▓░░░░░░░░░░░░  气泡 34%
  AIV_3  ▓▓▓▓▓▓░░░░░░░░  气泡 12%
  ```
  bar 的颜色跟随当前颜色模式（利用率用蓝/绿，气泡用橙/红）
- Bubble 标记：超过阈值的气泡自动标红，不需要用户手动开关
- Critical Lane 高亮：点"看瓶颈"按钮后，自动滚动到最差的那条 lane，聚焦视野

这一步的目的是让用户从 73 条 lane 里快速定位到 3-5 条值得深入的 lane，不用逐条扫描。

---

### 旅程 3：深入一个任务，找到原因

**用户问题**：这个 task 为什么这么长？它是哪个算子？

**当前体验**：Detail Panel 已经有字段，但呈现是纯表格，用户需要自己理解 rootHash / callOpMagic 是什么意思。

跳转功能（Open Pass IR / Open Source Flow）依赖 **program.json** 作为前置数据源。没有绑定 program.json 时，Open Pass IR 按钮处于 disabled 状态，用户没有任何提示说明原因。toolbar 上有"绑定 Program"按钮，task 详情里也有"Bind Program"按钮，但两者都没有解释为什么要绑定、绑的是什么文件。

#### 什么是 program.json？

`program.json` 是 PyPTO 编译器输出的 IR 序列化文件，记录了一次编译的完整计算图状态。它的核心作用是把三个视图串联起来：

- **源码视图**：每个 operation 上的 `semantic_label`（含文件名 + 行号），是用户在代码里调用 `pypto.set_semantic_label("label_name")` 打的标记
- **Execute Graph 视图**：每个 operation 的 `opmagic`（program.json 里的实际字段名），是全局唯一整数 ID；swimlane trace 的 event-hint 字符串里以 `callOpMagic` 的形式展示，两者对应同一个值
- **Block Graph（Pass IR）视图**：CALL operation 的 `calleehash`，指向被调用的 kernel 函数 hash

文件通常位于编译输出目录的根部（`output_xxx/program.json`），与各 Pass 的 Before/After JSON 同目录。加载 swimlane 文件时，工具会自动尝试从同目录加载 `program.json`（best-effort，失败静默）。

**理想叙事**：
- 点击 task 后，Detail Panel 不只是字段列表，而是情境化解读：
  - "这是第 3 个 stitch 里的 MatMul 算子（subgraph 0）"
  - "它的 inputRawMagic 有 3 个张量，其中一个在上一个 task 里刚写完（因果链）"
  - "对比 reference 数据集，这个 task 在 after 版本里缩短了 82%"
- 跳转按钮更清晰：Open in Execute Graph / Open in Source Flow，而不是 Pass IR 这种工具名
- 当 program.json 未绑定时，Open Pass IR 按钮应有明确提示（"需要先绑定 program.json，路径：output_xxx/program.json"），而不是静默 disabled
- **计算图分屏（iframe split-screen）**：点击 task 后，不跳转新标签页，而是在 swimlane 右侧内嵌一个 iframe 分屏，直接展示对应的计算图视图并高亮目标 op（见下方详细方案）
- **Task 因果链连线**：点击 task 打开详情面板的同时，在 lane 视图上叠加显示该 task 的前驱（in）和后继（out）task 连线，直观呈现 operand 的产生和消费关系。连线应跨 lane 绘制（输入可能来自不同的 AIC/AIV lane），点击连线端点可以跳转聚焦到对应的 task

#### 计算图分屏方案（基于现有架构）

当前实现是 `window.open()` 打开新标签页，两个视图完全独立。分屏方案的核心是把这个导航行为改为加载 iframe。

**现有基础（不需要大改）：**

| 模块 | 入口 | 当前聚焦机制 | Preview 模式 |
|------|------|-------------|-------------|
| Pass IR | `pass-ir/index.html?action=open-file` | sessionStorage bridge（传 `semanticLabel` + `callOpMagic`） | ✅ 已实现：`?preview=1` 触发 `body.preview`，隐藏所有 UI chrome |
| Source Flow | `source-flow/index.html` | sessionStorage bridge（传 `semanticLabel`） | ⚠️ 尚未实现，需补充一段 CSS + 激活逻辑 |

两个模块都用 sessionStorage 作为数据桥梁，**同源 iframe 与父页面共享 sessionStorage**，因此现有桥梁机制无需改动即可在 iframe 里生效。

**实现步骤：**

1. **source-flow 补充 preview 模式**：参考 pass-ir 已有的 `.preview` CSS，隐藏 header / panel / controls，只露出图形视区；通过 `?preview=1` 激活
2. **swimlane 布局改造**：swimlane 主视图右侧增加可折叠的 iframe 区域（CSS grid，可拖拽调整宽度）
3. **导航逻辑改造**：将 `openPassIrForTask()` 和 `openSourceFlowForTask()` 从 `window.open()` 改为：写入 sessionStorage bridge → 设置 iframe.src → 展示 iframe 区域
4. **iframe URL**：
   - Pass IR：`../pass-ir/index.html?action=open-file&preview=1`
   - Source Flow：`../source-flow/index.html?preview=1`

**布局结构设想见第四节（整体 UI 形态）。**

---

### 旅程 4：验证优化效果

**用户问题**：改完之后，到底好了多少？

**当前体验**：Before/After Diff 已有，但 delta summary 卡是分开的，用户需要自己对比两边的数字。

**理想叙事**：
- 同屏 Diff：before 在上，after 在下，同步滚动，相同 lane 对齐
- 改进标注：after 中比 before 短超过 30% 的 task，自动加绿色标记
- Diff Summary 卡：
  - "AIC 利用率 +12.3%，AIV 利用率 +55.1%"
  - "消除了 47 个气泡（共节省 463us）"
  - "MatMul 相关 task 缩短了平均 82%"

---

### 旅程 5：记录发现，沉淀分析

**用户问题**：这次分析怎么保存下来？下次能复现吗？

**当前体验**：没有持久化机制。

**理想叙事（Journal 核心功能）**：
- 任意时刻可以"拍一张快照"：保存当前的缩放、选中状态、measure 区间、marker
- 可以给快照加文字注释："这里是 AIV 气泡最集中的区间，对应 tiling 参数 vec_tile=128 时的碎片"
- 多个快照组成一篇 Journal：现象 -> 假设 -> 证据 -> 结论 -> 动作
- Journal 可以导出为 Markdown，带截图和参数

## 四、整体 UI 形态设想

### 整体布局原则

**主视图（swimlane 或 swimlane + 计算图分屏）始终占满整个窗口**，旅程引导面板以悬浮窗的形式覆盖在右侧，不压缩主视图的宽度。面板高度 = 窗口高度 - 上下 margin，内部独立滚动。

---

**旅程卡片结构（每张卡，Material Design compact 风格）：**
- 标题：当前步骤名
- 数据摘要：从已加载文件自动提取的关键指标（利用率 / 热点 lane / 最大气泡等）
- 推荐操作：1–3 个可点击快捷动作，点击直接触发对应功能
- 前置依赖（如有）：内嵌所需资源的加载入口，例如 Step 3 内嵌 program.json 绑定按钮：

绑定成功后按钮替换为文件名 + ✓ 状态标记，不再重复提示。

#### 状态二：Task 悬浮详情卡（in-place popup）

点击任意 task，**在 task 原位置就近弹出悬浮卡片**，而不是在右侧打开一个固定的 detail panel。卡片自动避让视口边缘（靠左侧的 task 向右弹出，靠右侧的向左弹出）。

```
+─────────────────────────────────────────────────────────────────+
│                                                                 │
│  lane: AIC_7  ██  ┌────────────────────────────────┐  ████    │
│               click│ MatMul_sg0  ·  stitch 2        │          │
│                    │ 时长 1.243ms  ⚠ vs ref +82%    │          │
│                    │ seqNo: 47  callOpMagic: 0x271a  │          │
│                    ├────────────────────────────────┤          │
│  lane: AIV_3  ▒▒▒▒│ 推荐操作                        │▒▒        │
│               ▒    │ [→ 在 Pass IR 中查看]           │          │
│                    │ [⊕ 显示前后依赖连线]            │          │
│                    │ [⇄  在计算图中分屏联动]         │          │
│                    │ [📌 添加到 Journal]             │          │
│                    └────────────────────────────────┘          │
│                                                                 │
+─────────────────────────────────────────────────────────────────+
```

旅程引导面板保持悬浮，不受 task 弹窗影响；两者可以同时存在。

#### 状态三：计算图分屏联动

用户点击"在计算图中分屏联动"（task 弹窗或旅程卡片均可触发），主视图分裂为上下两栏，各占全宽横向展开。旅程面板依然悬浮在右侧，此时内容可简化为只显示与计算图联动相关的操作。

```
+─────────────────────────────────────────────────────────────────+
│  Pass IR / Source Flow（iframe，全宽）              [关闭分屏 ×] │
│                                                                  │
│  ○──○──╔═══╗──○──╔═══╗                                         │
│         ║op1║     ║op2║←高亮                                    │
│         ╚═══╝     ╚═══╝                                         │
│  ○──○──────────────────○──○                                     │
├──────────────────── 拖拽分割线 ─────────────────────────────────┤
│  Swimlane（全宽）                                                │
│                                                                  │
│  lane: AIC_7  ██  [selected ▲]  ████                           │
│  lane: AIV_3  ▒▒▒▒▒  ▒                                         │
│                                                [timeline]        │
+─────────────────────────────────────────────────────────────────+
```

- **上栏**：Pass IR（计算图）或 Source Flow（计算流程），iframe 嵌入全宽，`?preview=1` 隐藏 UI chrome，图可以横向平铺
- **下栏**：Swimlane 泳道全宽展开，选中的 task 保持高亮
- 拖拽分割线可调整上下比例
- 点击上栏任意 op 节点，下栏对应 task 联动高亮（postMessage 协议，后续增强）
- 关闭分屏后恢复默认全屏泳道视图

---

### 三种状态的切换关系

```
默认状态（状态一）
    │
    ├── 点击 task ──────────────────► 状态二（浮动卡片，叠在泳道上）
    │       │
    │       └── 点击"分屏联动" ─────► 状态三（图 + 泳道分屏）
    │                                    │
    │                                    └── 关闭分屏 ──► 回到状态一
    │
    └── 点击旅程卡推荐操作 ──────────► 触发对应功能（排序/跳转/开启标红…）
```

每个 Journal 快照（旅程 5）保存的是**当前处于哪个状态、视图参数是什么**，回放时直接还原到对应状态。

## 五、优先级建议

当前工具的最大 ROI 提升点，不是加功能，而是**把现有功能组织进旅程**。

| 优先级 | 方向 | 改动量 | 价值 |
|--------|------|--------|------|
| P0 | 旅程引导面板：右侧悬浮卡片面板，Material Design compact 样式，每步一张卡（数据摘要 + 推荐操作） | 中（新增组件） | 极高：消除"不知道从哪看"的迷失感，把所有功能收进旅程叙事 |
| P0 | 全局健康卡放在旅程面板顶部：加载文件后自动填充利用率 / 气泡 / 一句话诊断 | 小（复用 buildExplanations，挂载位置改变） | 极高：第一眼就有分析入口 |
| P0 | Lane label 区域加 minibar（气泡/利用率） | 小 | 高：让排序有视觉感知，不只是文字 |
| P1 | Task in-place 悬浮弹窗：点击 task 在原位弹出，含情境化解读 + 推荐操作，替代当前固定 detail panel | 中 | 高：减少视线跳跃，技术字段变成可读结论，操作入口就近呈现 |
| P1 | 三状态 UI 架构（默认 / task 弹窗 / 上下分屏）及状态切换逻辑 | 中（布局重构） | 高：为所有联动功能提供统一的 UI 容器 |
| P1 | 计算图上下分屏（上栏 Pass IR / Source Flow iframe，下栏泳道，各占全宽，拖拽分割线） | 中（source-flow 补 preview 模式；pass-ir preview 已就绪） | 高：消除跳页断层，两个横向视图都不被压窄 |
| P1 | Task 因果链连线（点击 task 后在 lane 上叠加 in/out 连线，跨 lane 绘制） | 中 | 高：operand 依赖从数字变成可见因果图，辅助定位等待瓶颈 |
| P1 | Bubble 超阈值自动标红，无需手动开启 | 小 | 高：消除隐藏功能的发现成本 |

---

## 六、一句话总结

> Swimlane 现在是一个功能齐全的工具箱，需要变成一个带路的向导。
> 旅程引导面板是那个向导的形态：它不替用户分析，但它知道用户在哪一步、下一步该看什么，并把对应的功能直接递到用户手边。
