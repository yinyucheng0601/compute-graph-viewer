Touchpoint 02 · Memory & Transfer

内存体验的关键不是 API 名称，而是数据在哪里、何时移动、为什么慢

算子开发者最容易在 host/device 边界、global/shared/local 层级、coalescing、mask 和异步拷贝上犯错。优秀产品需要在写代码前提示风险，在 profile 后给出可执行改动。

01 Host / Device 边界

显式 copy、统一内存、框架 tensor 所在 device 需要一致解释，否则错误常在运行后才出现。

  • 风险：hidden copy
  • 机会：数据驻留图

02 访问合并

coalescing / transaction 是调优入门门槛。应从源码地址表达式推断风险。

  • 风险：stride 不连续
  • 机会：line-level warning

03 Shared / LDS

缓存 tile 可以提速，也可能引入 bank conflict、同步遗漏和占用下降。

  • 风险：过度缓存
  • 机会：tile advisor

04 Mask / Boundary

Triton 的 mask 降低越界风险，但也可能隐藏低效访问；CUDA/HIP 则依赖显式 if。

  • 风险：边界分支
  • 机会：mask coverage

Triton
offs = pid * BLOCK + tl.arange(0, BLOCK) x = tl.load(ptr + offs, mask=offs < n) tl.store(out + offs, x)
优势控制最细，Nsight 可观察 memory transactions、L2、shared throughput、bank conflict 等指标。API 迁移成本低，HIP 让 CUDA 用户快速进入 AMD GPU，但仍可触达底层计数器。地址和 mask 表达紧凑，block pointer 降低复杂 tile 访问样板代码。
劣势用户必须自己建立“地址表达式 → 事务 → cache → occupancy”的因果链。CUDA 风格代码迁移后，性能直觉不一定成立，wavefront/LDS 差异容易被忽视。编译器自动优化让内存行为更难预测，profile 异常时需要看 IR/后端。
适合场景极限调优、复杂数据布局、硬件特性充分利用。CUDA 代码迁移、多硬件支持、开放栈生产部署。tile 友好的 DNN 算子、快速探索、fused op。
总结最强但最重。迁移友好但需差异提示。最轻但需要解释编译器行为。

我们的方向：Memory Path Inspector

核心理念：每个内存建议都必须回到源码地址表达式

01
数据驻留图从 tensor/device/copy API 生成 host、device、cache、temporary buffer 的路径图，标出 hidden copy 和同步边界。
02
访问模式预检静态分析 idx/stride/mask，提前提示非连续访问、边界分支、可能的 bank conflict。
03
Profile 后行动建议每条建议包含证据指标、疑似地址表达式、最小改动、预期指标变化，避免“只给 counter”。
