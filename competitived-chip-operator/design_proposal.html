Product Proposal

把芯片 SDK 做成可解释的算子开发工作台

建议自研芯片/算子平台不要只对标 CUDA 的 API 覆盖度，而要对标开发者任务成功率：能否快速搭好环境、建立 mental model、写出正确算子、定位错误、解释性能瓶颈，并稳定接入框架和部署链路。

可诊断每个错误都有症状、证据、最短排查路径。
可解释每个编译和性能结论能回到源码、IR 或硬件事件。
可复现环境、profile、benchmark、autotune 结果可保存和比较。
可生产框架接入、CI 门禁、ABI 和性能回归进入默认流程。

六个核心功能模块

模块解决问题关键能力竞品参照优先级
Stack Doctor 环境和生态集成不确定 自动读取 driver/runtime/framework/GPU SKU，输出支持等级、风险和修复命令。 ROCm 兼容矩阵、CUDA 容器生态 P0
Kernel Error Explainer 错误报告只给后果，不给原因 非法访问、数值偏差、异步错误映射到源码行、坐标、shape/stride 和下一条命令。 Compute Sanitizer、Triton interpreter P0
Profile Experiment Profiler 指标不能转成行动 把每条性能建议表达为假设、证据、最小改动、预期指标变化和实验结果。 Nsight source correlation / baseline P0
Operator Lab 算子从正确到高性能缺少闭环 reference、kernel、随机测试、边界 shape、benchmark、profile、框架绑定模板一体化。 Triton Python-first / autotune P1
Compile Explanation编译器优化不可解释输出 tile、指令、寄存器、未触发优化和触发条件，关联源码与 IR。CUDA PTX/SASS、ROCm LLVM、Triton MLIRP1
Production Gate本地可跑但生产回归CI 中检查 correctness、edge shape、latency p95、memory footprint、ABI drift。CUDA 默认生态、ROCm 镜像策略P2
Recommended workflow
stack doctor -> create operator lab -> run correctness tests -> kernel error explainer -> profile experiment -> compile explanation -> autotune -> framework integration -> CI regression gate

落地路线

MVP：先降低失败率

完成 Stack Doctor、Operator Lab 模板、基础 correctness test、benchmark runner。目标是让新开发者在 30 分钟内跑通一个可验证算子。

V1：打通解释链路

接入 profiler、compile explanation、源码行映射和 profile baseline。目标是把性能问题从“看指标”推进到“做实验”。

V2：生产化闭环

加入 CI 性能门禁、版本矩阵、容器镜像、框架扩展模板和回归报告。目标是让算子进入真实产品链路。

核心张力追求 CUDA 兼容度 vs 建立自己的可解释体验
优化方向API 兼容只作为迁移入口，核心差异化应放在诊断、解释和工作台闭环。
局限性如果底层 profiler、compiler metadata、runtime trace 不开放，上层体验只能做到部分解释。

成功指标和验收口径

模块核心指标验收方式失败信号
Stack Doctor首次可运行时间、环境失败率扫描后输出状态、风险和修复命令；容器建议可直接复制。用户仍需要人工查版本表。
Error Explainer非法访问定位时间、reference diff 覆盖率错误报告包含源码行、坐标、疑似原因和下一条命令。只输出 runtime error，没有排查路径。
Profile Experiment实验可复现率、建议采纳率每次 profile 自动生成 baseline、shape、config、代码 hash 和结果。用户用截图和手工表格记录调优。
Operator Lab首个正确 fused op 时间模板自动生成 reference/test/benchmark/export。用户仍需从多个文档拼接工作流。
Production GateCI 拦截回归数量、升级失败率correctness、latency p95、ABI drift 都能在 CI 中失败。生产升级后才发现性能或 ABI 问题。
