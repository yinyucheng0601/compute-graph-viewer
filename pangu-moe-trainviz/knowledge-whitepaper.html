大模型训练并行放置白皮书
Rank / Device Placement 的语义边界与产品表达

本文面向 TrainScope 的并行放置建模，区分 device、rank、stage、expert 等对象，并说明 DP、FSDP / ZeRO、PP、TP、SP/CP、EP、VPP、micro-batch 与重计算各自作用在哪一层。核心结论是：产品界面应表达“多维运行时放置 + 通信后果”，而不是把所有语义压成“每张卡承载哪些层/算子”。

7+需要区分的策略：DP、TP、PP、EP、SP/CP、FSDP/ZeRO、VPP 等
8核心字段：device、rank、PP、TP、DP、CP/SP、EP、state shard
1核心判断：训练放置是多维 placement，而非单一层归属
3网络层：Scale-Up / Scale-Out / Frontend 各自承载不同通信流
核心结论：PP 才适合说“哪段层在哪个 stage”。TP、DP、FSDP / ZeRO、SP/CP、EP 应表达为 rank/device 的多维坐标，并在 runtime 面板展示 All-Reduce、All-to-All、P2P、All-Gather / Reduce-Scatter、KV / activation exchange 等通信事件。

按语义建模与产品落地顺序组织

报告先定义基础对象，再依据官方语义拆解各类并行方式，最后落到 TrainScope 的数据字段、视图层级和交互文案。目录按照“分析对象、设计意义、交付结论”组织。

01 官方结构 对齐不同并行方式的核心切分对象。 避免把不同并行语义统一误解为层切分。 建立并行方式与切分对象的总表。
02 通信映射 把 DP、TP、PP、EP、CP/SP 映射到通信原语和网络层。 解释“放置关系”在训练 step 里真正导致哪些数据移动。 形成通信原语、关键指标、Scale-Up / Scale-Out 放置建议。
03 策略层次 把 TP、PP、EP、DP、ZeRO、SP/CP 放到不同作用层次上。 明确 Kernel、图编译器、分布式规划器和运行时各自不需要“全管”。 形成端到端自动优化需要的 IR、reshard、通信、训练/推理语义与代价模型清单。
04 基本对象 区分 card、device、rank、worker、stage、expert。 对象层级决定后续图表、tooltip 和数据模型是否准确。 明确标题、tooltip 和 placementMap 应该落在哪些字段上。
05 DP（Data Parallel，数据并行） 说明完整模型副本如何处理不同数据 shard。 明确 DP（Data Parallel，数据并行）不表达层段归属。 区分数据 shard、样本、mini-batch tensor 与模型参数副本。
06 FSDP / ZeRO（状态分片） 说明参数、梯度、优化器状态如何分片与重建。 FSDP / ZeRO 表达训练状态管理，而非层归属。 明确 state shard、all-gather、reshard 的展示位置。
07 PP（Pipeline Parallel，流水线并行） 说明模型如何按连续层段切分为多个 stage。 PP（Pipeline Parallel，流水线并行）是层段归属表达的主要适用场景。 给出可用于 inspector 的层段范围文案。
08 TP（Tensor Parallel，张量并行） 说明同一层内部的算子和矩阵如何分片。 TP（Tensor Parallel，张量并行）表达 shard 关系，而非整层归属。 明确应展示 QKV shard 2/8，而非“负责 Attention 层”。
09 SP / CP（Sequence / Context Parallel，序列 / 上下文并行） 说明长序列和上下文窗口如何分片。 SP/CP（Sequence / Context Parallel，序列 / 上下文并行）解决 token 维度负载，而非层切分。 给出 sequence shard / context shard 的页面表达。
10 EP（Expert Parallel，专家并行） 说明 Gate 如何将 token 路由到不同 expert 与 rank。 MoE 场景中需要避免把 expert ↔ rank ↔ card 固定为 1:1 关系。 明确产品应展示 expert placement，而非 MoE 层归属。
11 多维叠加 说明 ParallelDemo 的 dp × pp × cp × tp 网格公式如何映射到 rank 坐标。 避免把“多维语义标签”误解成五张卡或单一卡片归属。 形成“公开模型配置 → 派生训练策略 → rank mesh → 单卡计算切片”的主图。
12 产品表达 说明并行数量是否可由开发者配置，以及盘古 MoE 单卡内部会呈现哪些计算。 这决定 TrainScope 的数据接口、聚合层级和单卡详情是否可扩展到真实卡数。 形成字段、tooltip、单卡详情结构与 MoE expert placement 建议。
Official Structure · 官方语义边界

并行方式的切分对象与语义边界

不同并行方式虽然都以“并行”命名，但其官方语义指向的数据对象并不相同。先建立切分对象总表，可以为后续图表和产品字段提供一致的判断依据。

并行方式 核心切分对象 最接近的官方语义 典型通信 / 网络敏感点 适合的页面表达 不推荐的误导表达
DP（Data Parallel，数据并行） 数据 batch shard 同一模型副本跑不同数据，再同步梯度 梯度 All-Reduce；关注 bucket、重叠、跨 HBD 带宽和分层归约 DP7 replica “这张卡负责第 12-24 层”
FSDP / ZeRO（状态分片） 参数 / 梯度 / 优化器状态 训练状态分片，前后向按需 all-gather / reshard All-Gather / Reduce-Scatter；关注显存窗口、状态重建和通信计算重叠 state shard 3/8 “卡 12 负责 MLP 层”
PP（Pipeline Parallel，流水线并行） 连续层段 / stage 子模块 每个 stage 运行模型的一部分 activation / gradient P2P send-recv；关注相邻 stage 路径和 pipeline bubble PP5 · layers 36-42 “TP2 负责第 36-42 层”
TP（Tensor Parallel，张量并行） 同一层内部的张量 / 算子分片 列切 / 行切的线性层并行 层内 All-Reduce / All-Gather / Reduce-Scatter / P2P；低时延和拓扑跳数敏感 QKV shard 2/8 “卡 7 独自承载 Attention 层”
SP / CP（Sequence / Context Parallel，序列 / 上下文并行） sequence / context shard 参数复制但输入序列被切开 attention 相关 KV / activation exchange；同时受网络、缓存和内存层级影响 sequence shard 1/4 “CP3 承载第 20 层”
EP（Expert Parallel，专家并行） experts / expert shards 每个 worker 持有一个或多个专家 MoE dispatch / combine All-to-All；关注热点专家、尾时延和拓扑感知路由 experts 152-159 “卡 19 负责整个 MoE 层”
章节组织原则：后续章节均按照“实现目标 / 设计理由 / 代码或配置落点 / 验证方法”展开，目标是将并行语义落实为 TrainScope 的界面字段、数据模型和 tooltip，而非仅给出概念定义。
Communication · 网络里发生了什么

并行 placement 最终会变成训练 step 里的数据移动

把这一节读成三件事：谁组成通信组、什么时候移动 tensor、落在哪个网络域。页面不需要把所有细节铺开，只要把通信事件挂回对应的并行坐标。

修正口径：PP / TP / DP / CP / EP 不是时间轴上的阶段。timeline 里真正出现的是 compute、P2P、All-Reduce、All-to-All、All-Gather、Reduce-Scatter、KV / activation exchange 等事件；事件再携带 pp/tp/dp/cp/ep 标签。
并行方式主要通信最敏感页面指标
DP 梯度 All-Reduce 带宽、bucket、overlap dp_group、bytes、overlap、跨域链路
FSDP / ZeRO 参数 All-Gather、梯度 Reduce-Scatter 显存窗口、预取、reshard state_shard、gather window、reshard point
TP 层内 All-Reduce / All-Gather / Reduce-Scatter 低时延、p99、拓扑跳数 tp_group、op shard、collective、wait
PP activation / gradient P2P send-recv 相邻 stage 距离、bubble pp_stage、send/recv bytes、idle time
EP / MoE dispatch / combine All-to-All 专家倾斜、热点链路、尾时延 expert_bucket、load、hotspot、drop
CP / SP KV / activation exchange 上下文长度、缓存、内存层级 token range、KV bytes、cache hit、wait
网络放置判断结论TrainScope 要回答的问题
TP / EP / 高频 CP 优先放在 Scale-Up / HBD 或低跳数拓扑内。 是否跨域拉长 step time？热点链路在哪里？
DP / FSDP / ZeRO 可跨 Scale-Out，但要看分层归约、bucket 和 overlap。 跨域带宽是否被占满？通信是否与计算重叠？
PP 相邻 stage 不要求全组同域，但相邻 stage 不应走高时延路径。 bubble 来自算力不均，还是链路等待？
Frontend 只承载管理、存储、日志和控制面流量。 训练关键路径是否误走前端网络？
保留重点：placement 决定“谁在同一组”，communication events 解释“网络里实际发生了什么”。调度器应输入并行组、拓扑域和通信算法，而不是只按空闲卡数做 bin packing。
Strategy Layering · 策略层次与组件边界

不同并行策略作用于不同层次，不是每个组件都要处理全部策略

Megatron Core 官方并行指南把 DP、TP、PP、CP、EP、FSDP 分别定义到 batch、individual layers、model depth、sequence length、MoE experts 和 model state 等对象上；MindSpeed / MindSpeed-LLM 官方特性列表也同时覆盖 DP、TP、PP、VPP、SP、长序列并行、EP、分布式优化器、重计算、融合算子和通信重叠。因此“训推级主动能力”不能只写成照顾 TP、PP、EP，而应先区分策略作用层次和组件职责边界。

策略主要作用范围主要责任
TP 层内、Tensor 维度 张量切分、局部计算、集合通信；典型对象是同一层内的 Column / Row Parallel Linear、词表分片和 logits 分片。
PP 层间、子图或 Stage Stage 划分、micro-batch 调度、相邻 stage 的 activation / gradient P2P 传递。
EP MoE 专家层 Expert 放置、Token 路由、dispatch / combine All-to-All，以及专家负载均衡。
DP 数据批次 模型副本、样本切分、梯度同步和梯度通信重叠。
ZeRO / FSDP 参数、梯度、优化器状态 状态分片、按需 All-Gather / Reduce-Scatter、reshard、优化器状态显存压缩。
SP / CP 序列或上下文维度 序列切分、上下文通信、KV / activation exchange；SP 常与 TP 配套。
MindSpeed 口径修正：MindSpeed 官方的并行策略和自动调优范围不只包含 TP、PP、EP，还包含 DP、SP、CP、ZeRO / 分布式优化器、VPP、重计算、micro-batch、通信重叠和融合算子等因素；这些配置会共同影响算子 shape、显存峰值、通信事件和调度顺序。
组件通常需要处理的范围不应强行承担的责任
普通 Kernel 处理当前 rank 上的局部 Tensor、局部 shape、dtype、stride、tiling 和访存流水。 不决定全局 rank mesh、stage 划分或跨 rank 通信组。
图编译器 处理 layout、融合、依赖关系、buffer 生命周期和必要的重切分 / reshard。 不独自决定训练集群的全局并行策略组合。
分布式规划器 处理设备放置、rank mesh、并行策略组合、通信组和拓扑约束。 不替代单个 kernel 的 tiling，也不替代 MoE runtime 的 token 路由。
PP 调度器 处理 micro-batch、stage 执行顺序、bubble、VPP interleaving 和 P2P send/recv。 不决定同一线性层的列切 / 行切方式。
MoE 运行时 处理 EP 路由、token bucket、专家负载、dispatch / combine All-to-All。 不负责普通 Dense 层的 TP 切分传播。
训练运行时 处理 Autograd、Saved Tensor、重计算、梯度累积、优化器状态和分布式优化器通信。 不把推理请求调度或 KV Cache 生命周期混入训练 step 语义。
推理运行时 处理 KV Cache、Prefill / Decode、连续批处理、请求调度和动态 batch。 不等同于训练中的 micro-batch 流水线调度。
端到端主动优化能力必要条件
统一 IR 需要同时表示算子、计算图、设备放置、Tensor Layout、通信组和状态生命周期。
切分传播与 Reshard 需要描述一个并行布局如何在层间、算子间或训练阶段转换到另一个布局，并明确重切分成本。
通信建模 需要显式表示 collective、P2P、All-to-All、通信计算重叠、拓扑域和通信库算法选择。
训练语义 需要覆盖 Autograd、Saved Tensor、重计算、梯度累积、参数 / 梯度 / 优化器状态分片。
推理语义 需要覆盖 KV Cache、Prefill / Decode、连续批处理、请求队列、动态 shape 和 cache eviction。
代价模型 需要同时估计计算、通信、显存、流水空泡、动态负载、专家热点和长上下文缓存成本。
合法性与回退机制 需要校验数值正确性、shape 约束、资源约束、拓扑限制和不支持场景的 fallback。
产品含义：TrainScope 可以在一个视图里关联这些策略，但数据模型应保留组件边界。普通 kernel 面板看局部 tensor 和 tiling；图编译面板看 layout / fusion / reshard；分布式面板看 rank mesh 与通信组；PP / MoE / 训练 / 推理运行时分别呈现自己的调度语义。
Scope · 基本对象

card、device、rank、worker、stage、expert 的层级关系

若产品界面仅使用“卡”作为统一描述，硬件位置、运行时身份、层段分工与专家放置容易被混合表达。本节先建立对象层级，作为后续 placement 建模的基础。

对象层级图 · 物理设备、运行时身份与并行坐标
读图顺序：左侧表示物理设备，中间表示运行时身份，右侧表示混合并行配置下的多维放置标签；这些标签是否全部出现，取决于实际训练配置。
Ascend / MindSpore 依据与边界：官方文档支持多种并行模式组合，配置项覆盖 data_parallel、model_parallel、context_parallel、pipeline_stage 和 MoE 专家相关字段。右侧“并行坐标层”是 TrainScope 的产品抽象；实际页面应按配置有则展示、无则隐藏。参考：分布式并行训练、配置文件说明。
命名原则：产品总标题建议使用 Runtime Placement、Rank / Device Placement 或“rank/device 并行放置关系”，避免使用“每张卡承载哪些层/算子”作为总括表达。后文涉及缩写时，统一按“缩写（英文全称，中文全称）”理解。
Parallel 01 · Data Parallel

DP（Data Parallel，数据并行）：完整模型副本跑不同数据，再做梯度同步

DP（Data Parallel，数据并行）的核心语义是多个 rank 持有同一份模型副本，各自处理不同的数据 shard，并在反向传播阶段同步梯度。它描述的是数据切分与梯度同步关系，而非模型层段切分。

DP 主图 · 多个老师拿同一本答案书，批不同学生作业
类比：多个老师拿着同一本答案标准批不同学生作业。这里“数据不同”指的是每个人拿到的样本集合不同，例如不同 mini-batch shard；不是说模型参数 tensor 长得不一样。
项说明
实现目标提升吞吐，让多个 rank 同时处理不同数据 shard。
设计理由DP 不切模型结构；数据 shard 指样本或 mini-batch 切分，关键是梯度同步。
通信 / 网络落点反向后做梯度 All-Reduce。重点看 bucket、backward overlap、跨 HBD 链路和 Scale-Out 拥塞。
代码或配置落点显示 DP replica id、sample range、batch shard、gradient sync group，而不是层段范围。
验证方法检查多个 DP rank 是否展示同样的模型结构描述，但数据 shard 不同；通信面板应反映梯度同步。
页面口径：DP7 replica 或“完整模型副本之一，处理当前 step 的样本 / mini-batch shard”是准确表达；“这张卡负责第几层”不属于 DP（Data Parallel，数据并行）语义。runtime 面板则应把 DP all-reduce 画成梯度同步事件，而不是画成样本数据回传。
Parallel 02 · Fully Sharded Data Parallel

FSDP（Fully Sharded Data Parallel，全分片数据并行）：训练状态分片与按需重建

FSDP（Fully Sharded Data Parallel，全分片数据并行）的重点是训练状态与显存管理：参数、梯度和优化器状态以分片形式保存，并在前向或反向传播的特定阶段按需 all-gather，再在使用后 reshard。

FSDP 主图 · 一本大书拆成多份，轮到某章时再拼起来
类比：一本很厚的参考书被拆成多份，老师们平时只背自己那一份；轮到某章要用时，才临时把那章拼齐。
项说明
实现目标降低显存占用，让更大的模型可训练。
设计理由FSDP（Fully Sharded Data Parallel，全分片数据并行）切分的是参数、梯度与优化器状态，这与“哪张卡负责哪一段层”属于不同问题。
通信 / 网络落点前后向窗口做参数 All-Gather，反向后做 Reduce-Scatter / reshard。重点看预取、显存窗口、带宽和 overlap。
代码或配置落点页面应展示 state shard、all-gather window、reshard point 等语义，而不是层标签。
验证方法检查同一模块在不同 rank 上是否展示成状态切片；前后向关键阶段是否存在 gather / reshard 事件说明。
页面口径：可以写“该 rank 持有模型参数/梯度/优化器状态的一部分分片；前后向期间按需 all-gather”，不要写“卡 12 负责 MLP 层”。通信视图应显示 all-gather window、reduce-scatter 和 reshard point，用于解释显存峰值与等待时间。
Parallel 03 · Pipeline Parallel

PP（Pipeline Parallel，流水线并行）：连续层段切分与流水执行

PP（Pipeline Parallel，流水线并行）将模型切分为多个连续 stage，每个 stage 运行一段连续层，micro-batch 沿流水线顺序推进。因此，PP 是最适合展示层段范围的并行方式。

PP 主图 · 工厂流水线按工序分站
layers 36-42、stage 5 这类文案在 PP 语义下是准确的。
类比：工厂流水线把一件产品拆成若干站，每一站只负责自己那段工序，上一站做完交给下一站。
项说明
实现目标把深模型按层段切开，让不同 stage 并行工作。
设计理由只有 PP 在官方语义上直接对应“这个 stage 运行模型的哪一部分”。
通信 / 网络落点相邻 stage 之间传 activation / gradient。重点看 send/recv bytes、相邻链路、micro-batch 数量和 bubble。
代码或配置落点适合在 placementMap 中直接保存 layerStart、layerEnd、stageId，并在 inspector 中显示层段范围。
验证方法切换 stage 时，界面必须显示连续层段；micro-batch 流转顺序应与 stage 顺序一致。
页面口径：PP5 · layers 36-42、“该 stage 执行第 36-42 层的前后向”这类说法是可靠的。通信面板还应显示该 stage 与前后相邻 stage 的 recv activation、send activation、recv grad、send grad 和 bubble time。
PP 与融合算子正确关系
PP 的边界 PP 划分 stage 并调度 micro-batch；它不是算子融合 pass。
融合算子的边界 融合发生在 stage 内，目标是减少访存和 kernel launch。
不能跨越的边界 跨 stage 要 send/recv activation / gradient，通常不能合成一个本地 fused kernel。
调度影响 Stage 切分要避开关键 fusion pattern，并同时考虑 activation 边界和 bubble。
VPP 影响 VPP 改变虚拟 stage 的交错顺序，不改变融合算子的 stage 内属性。
Parallel 04 · Tensor Parallel

TP（Tensor Parallel，张量并行）：同层张量分片与协同计算

TP（Tensor Parallel，张量并行）的核心语义是在同一层内部切分张量或线性层权重，由多个 rank 协同完成该层计算。它回答“该层如何分片”，而非“该层归属哪个 rank”。

TP 主图 · 一张大矩阵按列或按行分给多人同时计算
类比：四个人同时算同一页表格的不同列块。每个人都在参与这一层，但没有谁“独自拥有”这一层。
项说明
实现目标把单层内部的大矩阵与大算子切开，扩展单层可并行规模。
设计理由TP（Tensor Parallel，张量并行）的官方语义是行切 / 列切，体现的是同层协同计算，而非整层归属。
通信 / 网络落点TP 通信在层内关键路径上。重点看 collective 类型、等待时间、HBD 内低时延和拓扑跳数。
代码或配置落点适合显示 op name、tensor shard id、group size，如 Attention QKV shard 2/8。
验证方法切换到 TP（Tensor Parallel，张量并行）视图时，同一层应同时出现在多个 rank 上；说明文字必须强调 shard，而非层归属。
页面口径：卡 7 / rank_7 持有 Attention QKV 的 TP shard 2/8 是准确的；“卡 7 负责 Attention 层”会误导。runtime 视图应把 TP collective 标成同层事件，并显示 tp_group、collective 类型、等待时间和所在拓扑域。
对比项TP算子 tiling
所属层次 分布式并行策略，作用在同一层的全局张量、权重和 rank group 上。 单个 kernel 的本地实现策略，作用在当前 rank / device 上的局部 tensor 上。
切分对象 把权重、embedding、logits 或激活按列、行、词表或序列维分到多个 rank。 把本 rank 的局部 tensor 再切成 tile / block，用于访存和核内流水。
通信责任 需要插入 All-Reduce、All-Gather、Reduce-Scatter 等跨 rank 通信。 通常不创建跨 rank 通信，只决定本地数据搬运和计算循环形态。
先后关系 TP 先决定每个 rank 看到的局部 shape 和 layout。 tiling 再基于这个局部 shape 选择 tile 大小、访存路径和 kernel 内调度。
产品表达 应展示 tp_group、shard id、分片维度和 collective event。 应展示 local shape、tile shape、kernel 名称和本地耗时。
关键边界：TP 和 tiling 都在“切”，但前者是跨 rank 的张量所有权与通信语义，后者是单 rank 内 kernel 如何高效计算局部分片。图编译器可以在两者之间处理 layout、fusion、依赖和 reshard，但不能把 kernel tiling 误写成 TP 策略本身。
与 TP 强相关的算子 / 层典型 TP 关系页面表达建议
ColumnParallelLinear 按输出维切分；常见于 QKV、MLP up / gate projection。 QKV column shard 2/8、MLP gate shard、output all-gather。
RowParallelLinear 按输入维切分；常见于 Attention output 和 MLP down projection。 O-proj row shard、MLP down shard、output all-reduce。
VocabParallelEmbedding 词表维分片，每个 rank 持有一段 vocab embedding。 vocab range 64k-96k、embedding shard。
LM Head / VocabParallelCrossEntropy logits 按 vocab 维切分，loss 在 shard 间规约。 logits vocab shard、vocab-parallel CE、loss collective。
TP mapping / communication ops 负责 TP 区域边界上的 split / gather / reduce。 把它们标为 TP boundary collective，而不是普通 compute op。
LayerNorm / Dropout 的 Sequence Parallel 形态 TP 常配合 SP；LayerNorm、Dropout 可能触发 all-gather / reduce-scatter。 标成 TP+SP adjacent，避免说成单纯 PP 层归属。
MoE expert MLP / Grouped GEMM EP 是主语，但当 expert 内部 MLP 继续做 TP 时，expert linear 也会出现列切 / 行切和对应 TP 通信。 expert E17 · TP shard 1/4，同时保留 EP routing 和 TP shard 两组标签。
fused softmax / fused swiglu / fused layer norm / fused bias-gelu 这些是局部融合算子，本身不定义 TP 组；它们的输入 shape 和 layout 会被 TP / SP 分片影响。 在 kernel 面板展示 fused op，在 placement 面板展示 TP shard，二者通过 local shape 关联。
Parallel 05 · Sequence / Context Parallel

SP / CP（Sequence / Context Parallel，序列并行 / 上下文并行）：序列与上下文维度切分

SP / CP（Sequence Parallel / Context Parallel，序列并行 / 上下文并行）的核心语义是将 token 序列或上下文窗口切分为多个 shard，由多个 rank 分担长序列计算压力。它描述“哪个 rank 处理哪段 sequence/context”，而非层段归属。

SP / CP 主图 · 一本长小说被拆成连续片段阅读
类比：一本很长的小说被切成连续片段给不同人阅读，大家用同样的规则处理文字，但每人只处理自己那段 token。
项说明
实现目标在长上下文场景下把 sequence 维切开，降低单 rank 的序列负载。
设计理由SP / CP 回答“哪段 token 由哪个 rank 处理”，不是层段归属。
通信 / 网络落点Attention 相关 shard 会交换 KV、activation 或中间状态。重点看 token range、KV bytes、cache hit 和 attention wait。
代码或配置落点适合展示 sequence shard id、token range、context window。
验证方法同一层或模块在多个 SP/CP rank 上仍然存在，但 token range 不同；说明文字必须明确是序列切分。
页面口径：CP1 · sequence shard 1/4、“负责当前上下文窗口的一段 token”是准确的；“CP3 承载第 20 层”不准确。通信视图应额外标出 KV / activation exchange、cache 命中和 attention wait，否则长上下文瓶颈会被误解成普通层放置问题。
Parallel 06 · Expert Parallel

EP（Expert Parallel，专家并行）：专家放置与动态路由

EP（Expert Parallel，专家并行）的核心语义是专家放置与 token 路由：Gate 决定 token 被路由到哪个 expert，而 expert 再分布到多个 rank 上。它表达 expert placement，而非整个 MoE 层的单点归属。

EP 主图 · 医院分诊台把病人送到不同专科医生
类比：医院分诊台把病人送去不同专科医生。医生分散在不同诊室里，病人去哪个医生由 gate 决定，不是先固定绑卡再解释语义。
项说明
实现目标在 MoE 层内部把 experts 分布到多个 rank 上，提升专家总容量与并发。
设计理由EP 的主体对象是 experts；rank 持有 experts / expert shards，不是负责整个 MoE 层。
通信 / 网络落点核心是 MoE dispatch / combine All-to-All。重点看专家负载、热点链路、token bucket 和尾时延。
代码或配置落点适合在 placementMap 中展示 expertIds、expertShardRange、gate route target。
验证方法从任意 token 或 expert bucket 出发，页面能追到它被路由到哪个 rank/device；并且不假设 1:1 固定映射。
页面口径：EP19 · experts 152-159、“该 rank 持有当前 MoE 层的一部分 experts / expert shards”是准确的；“卡 19 负责 MoE 层”不准确。runtime 面板应显示 dispatch / combine All-to-All、expert load、hotspot link 和 token bucket，而不是只显示专家列表。
Framework · 多维并行叠加

真实训练不是单选题，而是多维坐标叠加

在混合并行训练中，同一个 rank 往往同时拥有 PP（Pipeline Parallel，流水线并行）、TP（Tensor Parallel，张量并行）、DP（Data Parallel，数据并行）、CP（Context Parallel，上下文并行）和 EP（Expert Parallel，专家并行）等坐标。这里的“叠加”不是把一张卡拆成多张卡，而是同一个 rank 在多个通信组、切分维度和专家路由维度中各有一个坐标。

与 ParallelDemo 的匹配关系：配置的并行度先生成 rank 网格，再把同一 rank 投影到不同并行坐标上。当前 Demo 用 world = dp × pp × cp × tp 推导 rank 和通信组；EP / MoE 作为专家放置与 token 路由语义叠加显示。
通信推导关系：rank mesh 只给出通信组成员关系，真正的网络事件由训练 step 触发。同一个 rank_299 / PP5 / TP3 / DP2 / CP0 / EP19 在一次 step 里可能先接收 PP activation，再执行 TP shard 计算，MoE 区域触发 EP All-to-All，反向传播后进入 DP All-Reduce；这些是 runtime event 的时间顺序，不是把 PP / TP / EP 画成互斥模式。

读图顺序：先看左边的 rank 身份，再看右边叠加的五类坐标。真正的 placement 是多维标签交叉，而不是单个归类。

概念在训练中的含义在图表中的表达边界
stage PP（Pipeline Parallel，流水线并行）把整网层序列切成的流水线阶段。 Stage 0 表示模型开头的层段，不是第 0 个训练 step；只有 Stage 0 才通常包含 Embedding。
训练样本 / sample 一条训练数据；语言模型里通常是一段 token 序列。 DP（Data Parallel，数据并行）下不同 rank 吃不同 sample shard，不是模型结构不同。
batch / mini-batch shard 一个 step 输入的一组样本；被 DP/CP 等切分后，每个 rank 只处理其中一部分。 可显示 samples 512-1023 或 mini-batch shard，不要把它说成 tensor shard。
tensor 数据、权重、激活、梯度在计算机里的多维数组表示。 TP（Tensor Parallel，张量并行）切的是权重/激活等 tensor 维度，例如 QKV / Out 第 4/8 片。
step 一次训练迭代，通常包含 forward、loss、backward、通信同步、optimizer update。 模型架构图只覆盖 forward 主干；step runtime card 才表达 backward、通信和参数更新。

术语准确含义产品口径
block / decoder block Transformer 里重复堆叠的完整计算单元，通常包含 Attention、FFN/MoE、Norm、Residual 等子结构。 用于 PP（Pipeline Parallel，流水线并行）层段范围：Decoder blocks 20-23。
layer 语境依赖词。它可以指完整 decoder block，也可以指 Linear、Attention、LayerNorm 等更小子层。 图里可兼容写 layers 20-23，但正文和 tooltip 要解释为 decoder blocks 20-23。
num_hidden_layers 配置字段。Hugging Face 官方口径中它表示模型的 block 数量。 Pangu 配置中的 num_hidden_layers=61 应解释为 61 个 decoder/Transformer blocks。
hidden state 每个 token 在模型内部的中间向量表示，典型形状是 [batch_size, seq_len, hidden_size]。它是 runtime tensor，不是模块。 用于激活、层间传递、profiler tensor 说明，例如 hidden state [batch, seq, 7680]。
hidden_size hidden state 每个 token 向量的宽度，也就是最后一维大小。 Pangu 配置中的 hidden_size=7680 表示向量宽度，不表示层数。

问题细化结论
这些并行度能否由开发者配置 可以配置，但受设备数、层数、head 数、序列长度、batch、拓扑和通信成本约束。ParallelDemo 的滑块表达配置空间，不表示任意组合都合法。
与 ParallelDemo 的公式关系 ParallelDemo 按 world = dp × pp × cp × tp 推 rank mesh。TrainScope 应复用坐标和通信组推导；若有独立 EP，需新增 EP 维度。
图表应该怎么展示 按三层展示：host/device → globalRank/localRank → pp/tp/dp/cp/ep。单卡面板再展开本 rank 的计算切片。
通信视图应该怎么展示 派生 communicationEvents：记录 type、group、bytes、latency、overlap、topology、source / target。
MindSpeed / Megatron 官方范围对 TrainScope 配置与调优视图的影响
并行策略不只 TP / PP / EP 配置面板应覆盖 DP、TP、PP、VPP、SP、CP、EP、ZeRO、micro-batch，并校验 world size、head、层数、序列和 expert。
重计算与显存优化会改变调度 重计算、checkpointing、参数预取和状态分片会改变 saved tensor、显存峰值和通信窗口。
融合算子与通信重叠会改变观测事件 fused ops、overlap、MC2 / CoC 会改变 kernel 粒度和瓶颈归因。
自动调优需要多目标代价模型 调优结果要解释 shape、显存、通信、bubble、专家负载和请求调度，不只给 TP/PP/EP 三个数。
并行策略对单卡 / 单 rank 内部呈现的影响以盘古 MoE 训练视图为例
PP（Pipeline Parallel，流水线并行） 决定这个 rank 处在第几个流水线 stage，以及该 stage 覆盖哪些连续层段。 显示 PP5 · layers 36-42、stage idle、send/recv 和 bubble。
TP（Tensor Parallel，张量并行） 决定同一层内部的张量 / 权重 / 算子切片，通常需要同组 all-reduce、all-gather 或 reduce-scatter。 显示 QKV、O-proj、MLP、Embedding、LM Head 的 shard id 和 collective。
CP（Context Parallel，上下文并行） 决定 attention 看到的 token / context 分片，以及 KV、QKV 或 attention 中间结果在组内如何交换。 显示 CP1 · token shard、KV exchange、cache hit 和 attention wait。
DP（Data Parallel，数据并行） 决定这个 rank 处理哪份样本 / mini-batch shard，并决定 backward 后的梯度同步组。 显示 sample range、batch shard、gradient all-reduce 和 overlap。
EP（Expert Parallel，专家并行） / MoE 决定当前 rank 持有哪些专家、token 经 gate 路由到哪些专家，以及 dispatch / combine 的通信路径。 显示 experts、token bucket、expert load、dispatch / combine All-to-All。

Findings · 总结

把这件事记成四条判断

如果只保留最重要的结论，应该是下面四条。

核心判断真实大模型训练不是“卡按层和算子分配”一句话能说完，而是多个并行维度共同定义 placement。
准确边界“哪张卡负责哪几层”只适用于 PP（Pipeline Parallel，流水线并行）；TP、DP、FSDP/ZeRO、SP/CP、EP 都需要更精确的语义字段。
产品方向TrainScope 的正确标题是 placement，不是 layer ownership；ParallelDemo 提供的是计算引擎，不是最终 UI 结构。
网络方向placement 必须继续推导 communication events：TP / EP 偏 Scale-Up，DP / 分层归约偏 Scale-Out，PP 关注相邻 stage，CP/SP 还要联动 KV cache 和内存层级。
最终结论：对 PP（Pipeline Parallel，流水线并行） 来说，“每张卡承载哪些层”基本准确，但还要补充 activation / gradient P2P、bubble 和 stage 内融合边界；对 TP（Tensor Parallel，张量并行） 来说，这句话只说对一半，真正要看同层 shard、高频同步，以及 TP 与本地算子 tiling 的边界；对 DP（Data Parallel，数据并行）/ FSDP / ZeRO / CP（Context Parallel，上下文并行） 来说基本不准确；对 EP（Expert Parallel，专家并行） 来说，应改成“该 rank/device 持有哪些 experts 或 expert shards，并在 runtime 里发生了哪些 dispatch / combine All-to-All”。
