竞争产品基本介绍与分析定位

这三类产品不是同一层级，但正好覆盖开发者会拿来比较的三种体验：完整 GPU SDK、开放替代栈、面向 DNN 算子的高生产率 DSL。后续所有触点都围绕“任务成功率”比较，而不是单纯比较 API 覆盖度。

CU

CUDA

定位：NVIDIA GPU 全栈开发平台。
用户：GPU kernel 工程师、框架团队、HPC/AI 基础设施团队。
任务：写 kernel、调试、Profiler 调优、接入框架和部署生态。
竞争角色：成熟基线和事实标准。

RO

ROCm / HIP

定位：AMD 开放 GPU 软件栈和 CUDA 风格迁移路径。
用户：平台团队、框架团队、需要 AMD GPU 支持的工程团队。
任务：CUDA 迁移、双栈维护、rocprof/rocgdb 诊断。
竞争角色：直接替代竞品。

TR

Triton

定位：面向 DNN 自定义算子的 Python DSL 和编译器。
用户：AI 研究员、模型优化工程师、框架工程师。
任务：快速实现 fused op、matmul、attention、layernorm。
竞争角色：跨界标杆。

Product Architecture Comparison

CUDA、ROCm/HIP、Triton 的产品架构对比

这一页比较的是三类产品如何把“用户代码”连接到“GPU 硬件执行”。CUDA 是完整 GPU SDK，覆盖从语言、库到工具和硬件的闭环；ROCm/HIP 是 AMD 的开放 GPU 软件栈，重点提供 CUDA 风格迁移路径；Triton 是面向深度学习自定义算子的 Python DSL，重点提升写算子和调参数的效率。

CUDA stack

CUDA：NVIDIA 的完整 GPU 开发平台，开发者可以从 CUDA C++ 写 kernel，也可以通过 cuBLAS、cuDNN、NCCL 等库和 Nsight 工具链完成调试、调优和生产部署。

视觉证据 01：官方文档如何组织学习入口

芯片 SDK 的第一界面通常不是 IDE，而是官方文档。文档的导航结构会直接影响开发者能否快速找到 programming model、安装兼容、调试工具、Profiler 和 API 参考。

CUDA Programming Guide 13.1
CUDA 的文档入口把学习路径拆成基础概念、CUDA C++、高级特性和技术附录。它的优势是“从入门到专家”的层级完整，但用户也容易被大量概念淹没。
ROCm Documentation 7.2.2
ROCm 首页把兼容矩阵、安装、AI/HPC 和系统优化放在显著位置，说明“是否支持当前硬件和框架组合”本身就是核心体验问题。
Triton Documentation
Triton 把 tutorials、Python API、Gluon、MLIR dialects 和 debugging 放在同一入口，强调 Python 工作流和编译器能力是同一个产品体验。

视觉证据 02：Profiler 需要把指标变成行动

性能分析工具不能只展示计数器。成熟的 Profiler 要回答“哪一行代码造成了这个指标”“和上一次基线结果相比变化多少”“下一步最小改动是什么”。

Nsight Compute documentation
Nsight Compute 文档明确强调交互式性能分析器、API 调试器、图形化报告、基线对比和命令行入口。这说明 CUDA 的性能工具不仅有指标，还在组织完整的调优工作流。

触点 01 · 架构理解

架构理解不是总图，而是代码如何落到硬件的解释链

开发者第一次写 kernel 时，需要同时理解执行层级、内存层级、编译层级和工具层级。竞品差异不在“有没有架构图”，而在能否把一个具体算子展开成可预测的执行故事。

01 概念入口

CUDA 用 grid/block/thread 建立 SIMT 心智；ROCm 借 HIP 降低迁移门槛；Triton 从 program/block/tile 入口减少线程细节。

优势标准心智最稳定，源码、PTX、SASS、Profiler 指标和硬件行为能形成解释闭环。

劣势概念栈厚，用户容易先被 stream、memory、occupancy、PTX 等名词淹没。CUDA-like 容易制造“等价错觉”，wavefront、LDS、工具链差异需要额外显性化。底层行为被编译器隐藏，性能异常时需要重新理解 IR、layout 和 backend。
一句话结论最强解释链，但入门路径要任务化。迁移路径友好，但差异诊断必须产品化。高生产率入口，但需要透明的编译解释层。

触点 02 · 内存与数据搬运

内存体验的关键不是 API 名称，而是数据在哪里、何时移动、为什么慢

算子开发者最容易在主机端/设备端边界、全局内存/共享内存/本地存储层级、访存合并、边界 mask 和异步拷贝上犯错。优秀产品需要在写代码前提示风险，在性能分析后给出可执行改动。

01 主机/设备边界

显式 copy、统一内存、框架 tensor 所在 device 需要一致解释，否则错误常在运行后才出现。

  • 风险：隐式数据拷贝
  • 机会：数据驻留图

02 访问合并

访存合并和内存事务是调优入门门槛。产品应能从源码地址表达式推断潜在风险。

  • 风险：stride 不连续
  • 机会：源码行预警

03 Shared / LDS

缓存 tile 可以提速，也可能引入 bank conflict、同步遗漏和占用下降。

  • 风险：过度缓存
  • 机会：tile advisor

04 Mask / Boundary

Triton 的 mask 降低越界风险，但也可能隐藏低效访问；CUDA/HIP 则依赖显式 if。

  • 风险：边界分支
  • 机会：mask coverage

总结最强但最重。迁移友好但需差异提示。最轻但需要解释编译器行为。

从前面的旅程和触点分析可以收敛出一条建设顺序：先解决环境、调试、调优三类最容易阻断任务的问题，再提升算子开发效率、编译解释能力和生产集成能力。

优先级模块解决的旅程低谷竞品证据阶段目标
P0Stack Doctor环境搭建、版本矩阵、ABI 重建ROCm 首页把兼容矩阵前置；CUDA 容器/Toolkit 生态成熟把文档矩阵变成可执行检查器

P0 依赖关系

Stack Doctor 是所有模块前置；Error Explainer 依赖源码映射；Profile Notebook 依赖基线结果存储。

P0 首版边界

不做完整 IDE；先覆盖 CLI/HTML 报告、JSON 输出和可复制命令。

P1 生产率

Operator Lab 和 Compile Explanation View 负责把 reference、autotune、IR、profile 接起来。

P2 生产化

Production Gate 把 correctness、edge shape、latency p95、ABI drift 放进 CI。

核心结论

总体判断：自研芯片 SDK 的差异化机会不是“比 CUDA 更像 CUDA”，而是在 CUDA 的成熟解释链、ROCm 的可迁移开放性、Triton 的高生产率工作流之间，做出可诊断、可解释、可复现、可生产的算子开发工作台。

产品能力优势 完整解释链：源码、编译产物、Profiler 指标、调试工具、库和框架生态能相互印证。 迁移入口友好：HIP 保留 CUDA 风格，ROCm 开放栈和 LLVM 生态利于供应链多元化。 高生产率入口：Python-first、@triton.jit、tile/block 抽象、autotune 让 DNN 自定义算子实验速度显著提升。
体验风险与边界 概念栈厚、供应商绑定强，新用户容易先被 stream、memory、occupancy、PTX/SASS 淹没。 “CUDA-like”容易制造等价错觉，wavefront、LDS、gfx 支持、版本矩阵和框架组合会放大落地风险。 不是完整 SDK，底层 profiler、backend、IR 和生产部署仍依赖 CUDA/ROCm/框架生态；自动优化也需要解释。
可迁移设计原则 自研平台不能只追 API 覆盖度，必须优先做源码行关联、基线对比、错误解释和一键可复现环境。 兼容 API 只解决上手，必须同步提供 Stack Doctor、迁移差异诊断和硬件模型解释。 自研平台应提供 Operator Lab：reference、测试、benchmark、autotune、profile、export 一体化，而不是让用户从零拼工具链。
