Touchpoint 01 · Mental Model

架构理解不是总图，而是代码如何落到硬件的解释链

开发者第一次写 kernel 时，需要同时理解执行层级、内存层级、编译层级和工具层级。竞品差异不在“有没有架构图”，而在能否把一个具体算子展开成可预测的执行故事。

01 概念入口

CUDA 用 grid/block/thread 建立 SIMT 心智；ROCm 借 HIP 降低迁移门槛；Triton 从 program/block/tile 入口减少线程细节。

竞品横向比较：mental model 的组织方式

维度

硬件协同的完整 SIMT 模型
ROCm / HIP
CUDA 风格入口 + AMD 差异解释

block/tile 级数据流模型
界面/证据
CUDA mapping
kernel<<<grid, block>>> blockIdx.x → row tile threadIdx.x → lane work shared[] → reusable tile
HIP mapping
hipLaunchKernelGGL(...) threadIdx/blockIdx 保留 wavefront=64 LDS 替代 shared mental model
Triton mapping
@triton.jit pid = tl.program_id(0) offs = pid*BLOCK + arange tl.load / tl.store with mask
优势标准心智最稳定，源码、PTX、SASS、Profiler 指标和硬件行为能形成解释闭环。
迁移入口熟悉，HIP 保留大量 CUDA 语法，可降低组织切换成本。
把 DNN 算子从逐线程上移到 tile 数据流，初始认知负担低。
劣势概念栈厚，用户容易先被 stream、memory、occupancy、PTX 等名词淹没。CUDA-like 容易制造“等价错觉”，wavefront、LDS、工具链差异需要额外显性化。底层行为被编译器隐藏，性能异常时需要重新理解 IR、layout 和 backend。
一句话结论最强解释链，但入门路径要任务化。迁移路径友好，但差异诊断必须产品化。高生产率入口，但需要透明的编译解释层。

我们的方向：代码到硬件的任务切片解释器

核心理念：每个示例都能展开为 execution story

01
示例旁固定放“执行展开”对每个 kernel 展示数据分块、并行粒度、内存路径、同步点和最可能瓶颈，不让用户在总图和源码之间跳转。
02
提供 CUDA 迁移术语表把 warp/wavefront、shared/LDS、block/program、stream/queue 拆成“相同、相似、危险不同”三类。
03
架构图按任务过滤用户选择 reduction、matmul、copy、attention 后，只高亮相关硬件层级和工具入口，减少首屏噪音。
